# インセプションデッキ — GitHub Issue Shortcut

> 作成日: 2026-07-10。前提: [プロジェクトミッション](../project-mission.md) / [リーンキャンバス](01-lean-canvas.md) / 各リサーチ（[市場・競合](../research/2026-07-10-market-competitors.md) / [モバイル UX・PWA](../research/2026-07-10-mobile-ux-pwa.md) / [認証](../research/2026-07-10-auth-architecture.md) / [Cloudflare スタック](../research/2026-07-10-cloudflare-stack.md)）。

## 1. 我々はなぜここにいるのか

- **「思いついた瞬間」を逃さないため**。タスク管理はアプリを起動するのすら面倒になったら終わる（[市場リサーチ §3](../research/2026-07-10-market-competitors.md) の一次証言）
- GitHub Issues で個人タスクを管理する文化は確立しているのに、**モバイルの即時キャプチャだけが欠けている**（公式アプリは 4 画面、自作解は PAT 管理が壁）
- Android×PWA のワンタップ起票は **空白地帯**（[市場リサーチ §1](../research/2026-07-10-market-competitors.md)）。まず開発者本人が毎日使う実用ツールとして作り、OSS として一般公開する
- あわせて、Claude Code による自律スプリント開発でプロダクトを完成させる実践の場でもある

## 2. エレベーターピッチ

> **思いついた瞬間に GitHub Issue を起票したい** Android 使いの開発者向けの、**GitHub Issue Shortcut** というプロダクトは、**ホーム画面から数秒で Issue を起票できる Web アプリ（PWA）** です。これは **PAT 管理不要（GitHub ログインのみ）で、ショートカットがリポジトリ / ラベルを覚えてくれる最速・最短の起票体験** を提供でき、**GitHub 公式アプリ（3〜4 タップ + 4 画面）や PAT 手動管理が必要な HTTP Shortcuts** とは違い、**セットアップ 0 分・起動 → 3 タップ以内・10 秒以内の起票** が備わっています。

## 3. パッケージデザイン

**製品名**: GitHub Issue Shortcut

**キャッチコピー案**:

1. 「思いついた瞬間、もう Issue になっている。」
2. 「ホーム画面から 3 タップ。GitHub 起票の最短ルート。」
3. 「PAT はもういらない。ログインして、置くだけ。」

**訴求ポイント（箱の裏面）**:

- ホーム画面のショートカットから起動即入力、10 秒以内に起票完了
- GitHub ログインのみでセットアップ完了（PAT の発行・管理・失効対応から解放）
- 送信失敗でも入力は消えない（下書き保全）
- インストール不要の PWA。将来は Play ストアからも（TWA）

## 4. やらないことリスト

| 区分 | 項目 | 理由 |
|---|---|---|
| **やる** | Issue 作成の最短フロー（タイトル + 本文） | コア体験そのもの（MVP） |
| **やる** | GitHub App 認証（PAT 不要） | セットアップ 0 分の要（MVP） |
| **やる**（次マイルストーン） | ラベル指定・ショートカット起動（リポジトリ / ラベルプリセット） | 価値提案の柱だが MVP からは分離 |
| **やらない**（当面スコープ外） | Issue 一覧・編集 | 閲覧・管理は公式アプリ / Web で足りる。「あってもいい」程度で優先しない |
| **やらない**（優先度低） | アサイン機能 | 個人利用が主で自分 = 担当が既定。push 権限がないと silently dropped される制約もあり（[認証リサーチ §6](../research/2026-07-10-auth-architecture.md)） |
| **やらない** | iOS 向け最適化（ブラウザでの基本動作は妨げない・NFR-16 参照） | iOS はネイティブ競合が既に 3 種存在。Android×PWA の空白地帯に集中 |
| **やらない** | 収益化（課金・広告） | 相場観から実用ツール / OSS 位置づけと決定（[市場リサーチ §4](../research/2026-07-10-market-competitors.md)） |
| **あとで決める** | オフラインキュー（Background Sync）・スマート入力・ウィジェット | [UX リサーチ §7](../research/2026-07-10-mobile-ux-pwa.md) の M3 以降。需要が立ってから |

## 6. 解決案を描く

技術スタックの全体像（詳細は [Cloudflare リサーチ §7](../research/2026-07-10-cloudflare-stack.md)）。

```text
[Android Chrome / ホーム画面ショートカット]
        │ HTTPS
        ▼
Cloudflare Workers（単一 Worker）
├── static assets: Vite + React SPA（PWA: vite-plugin-pwa / Workbox）
├── API: Hono（/api/*, /auth/*）
│     ├── GitHub OAuth（GitHub App user token・PKCE + state）
│     └── Issue 作成プロキシ（POST /repos/{owner}/{repo}/issues）
├── D1: セッション・暗号化トークン・ユーザー設定
└── Secrets: GitHub App client secret・トークン暗号鍵
```

- **認証**: GitHub App（Issues: write の最小権限）+ フルページリダイレクトの OAuth。トークン交換・保管はサーバー側（[認証リサーチ §7](../research/2026-07-10-auth-architecture.md)）
- **UX**: 起動即入力画面（app shell precache でサブ秒表示）+ 楽観的 UI + 失敗時下書き保全（[UX リサーチ §4–5](../research/2026-07-10-mobile-ux-pwa.md)）
- **段階拡張**: MVP（起票最短）→ M2（shortcuts + URL パラメータ + Share Target）→ M3（スマート入力・オフラインキュー）→ M4（TWA で Play 配布）

## 7. 夜も眠れない問題

| リスク | 影響 | 備え |
|---|---|---|
| GitHub 公式がモバイル起票 UX を改善 | 存在意義の低下 | 2025 年の新 Issue UI 改善でもモバイル Web への言及なし（[市場リサーチ §5](../research/2026-07-10-market-competitors.md)）。個人用ショートカット特化の優位は残る |
| App 未インストールのリポジトリに起票できない（GitHub App の制約） | 初回体験の躓き | オンボーディングで App インストールへ誘導する UX で吸収（[認証リサーチ §7](../research/2026-07-10-auth-architecture.md)） |
| リフレッシュトークン単回使用の競合失効 | 認証切れで起票失敗 | ユーザー単位の直列化設計（[認証リサーチ §2](../research/2026-07-10-auth-architecture.md)） |
| キーボード自動表示が PWA で不可能 | 「起動即入力」の理想に届かない | 最初のタップをジェスチャに使う緩和策で「起動 → 1 タップ → キーボード」（[UX リサーチ §2](../research/2026-07-10-mobile-ux-pwa.md)） |
| 再送による重複起票（API に冪等性キーなし） | 信頼性 KPI 毀損 | 直近送信内容の記録による自前の重複防止（[認証リサーチ §6](../research/2026-07-10-auth-architecture.md)） |
| 市場自体が狭い | 広がらない | 開発者本人が毎日使えれば最低限成功。OSS 公開で価値を残す |

## 8. 期間を見極める

[UX リサーチ §7](../research/2026-07-10-mobile-ux-pwa.md) のロードマップをマイルストーンとして採用する。AI エージェントの自律スプリント開発のため、期間はカレンダー時間でなく **スプリント数の目安**（較正前の粗い見積もり・[ミッションの SP 標準値](../project-mission.md) 準拠）。

| マイルストーン | 内容 | 目安 |
|---|---|---|
| MVP | GitHub App 認証 + 起票最短フロー + PWA インストール可能 + デプロイパイプライン（基盤整備 M0 を含む） | 数スプリント（最優先） |
| M2 | manifest shortcuts・URL パラメータ起動・Web Share Target・ラベルプリセット | MVP 後の 1 マイルストーン |
| M3 | スマート入力・ボトムシート UI・オフラインキュー | 需要とドッグフーディング結果次第 |
| M4 | TWA 化・Play ストア配布 | M2 安定後 |

- **MVP の完成条件**: 開発者本人がホーム画面から日常的に起票し、[KPI](../project-mission.md)（10 秒・3 タップ・99%・5 分セットアップ）を実機で満たす
- 見積もりは実測（tokens/sp）4 週分で較正する前提の暫定値

## 9. 何を諦めるか（トレードオフスライダー）

| スライダー | 優先度 | 方針 |
|---|---|---|
| 起票の速さ・確実さ | **最大（動かせない）** | 起票フローに 1 タップでも足す機能は既定でオフ。失敗時も入力を失わない |
| セットアップの容易さ | 高 | GitHub ログインのみ・5 分以内に初起票 |
| セキュリティ | 高（妥協しない） | 最小権限・トークン暗号化保管・RFC 9700 準拠（[認証リサーチ §3](../research/2026-07-10-auth-architecture.md)） |
| 機能の豊富さ | **低（諦める）** | 一覧・編集・アサインはやらない。表現力（ラベル等）は速さの次 |
| UI のリッチさ | 低（諦める） | 起動速度優先。バンドルサイズ・初期ロードを常に監視 |
| 対応プラットフォームの広さ | 低（諦める） | Android Chrome に集中。iOS・デスクトップ最適化はしない |

## 10. 何がどれだけ必要か

| リソース | 内容 | 費用感 |
|---|---|---|
| チーム | 開発者本人（オーナー / レビュー）+ Claude Code（自律スプリント実装） | AI 実行コストが主 |
| インフラ | Cloudflare Workers + D1（無料枠で運用可・[Cloudflare リサーチ §1, §3](../research/2026-07-10-cloudflare-stack.md)） | 0 円〜 |
| 外部登録 | GitHub App 登録（無料）・独自ドメイン（任意）・将来の Google Play 開発者登録（$25 一回・M4 時点で判断） | ほぼ 0 円 |
| 開発環境 | wrangler v4・Vite・vitest-pool-workers・GitHub Actions（CI/CD） | 無料 |
| 検証機材 | Android 実機（開発者所有・KPI の実機計測に必須） | 追加費用なし |

金銭コストはほぼゼロで、**主たる投資は AI エージェントの実行コストと開発者のレビュー時間**。収益ゼロ前提でも維持できる構成にする。
