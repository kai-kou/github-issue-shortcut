# ユーザーストーリーマップ — GitHub Issue Shortcut

> 作成日: 2026-07-10。バックボーン（ユーザー活動）→ ステップ → ストーリーの 3 層 + マイルストーン横断スライスで構成する。
> マイルストーンごとのゴール・Done 判定・SP 見積もり・リスクは [04-milestones.md](04-milestones.md)、機能要件 ID との対応は [00-requirements.md](00-requirements.md) を参照。
> 根拠: [ミッション](../project-mission.md) / [市場・競合リサーチ](../research/2026-07-10-market-competitors.md) / [モバイル UX・PWA リサーチ](../research/2026-07-10-mobile-ux-pwa.md) / [認証リサーチ](../research/2026-07-10-auth-architecture.md)。

## 0. 前提

- **主ペルソナ**: GitHub Issues で個人プロジェクト / 生活タスクを管理する Android 使いの開発者（第 1 号は開発者本人）。以下ストーリーの「開発者」はこのペルソナを指す（[市場リサーチ §3](../research/2026-07-10-market-competitors.md)）
- **中心となる利用シーン**: 移動中・家事中・別作業中に「思いついた瞬間」、ホーム画面から数秒で起票してすぐ元の文脈に戻る
- **KPI**（[ミッション](../project-mission.md)）: 起票 10 秒以内（タイトルのみ 5 秒以内）・ショートカット起動時 3 タップ以内・起票成功率 99%（失敗時も入力を失わない）・初回セットアップ 5 分以内

## 1. バックボーン（活動 → ステップ）

利用の時系列に沿った 4 つのユーザー活動と、その下のステップ。

| 活動 | A. セットアップする | B. 思いつきを起票する | C. ショートカットを整える | D. 起票後を確認する |
|------|-------------------|---------------------|------------------------|-------------------|
| 頻度 | 初回のみ（+ リポジトリ追加時） | 毎日・1 日複数回（コア体験） | ときどき（運用が固まるまで） | 起票直後・気が向いたとき |
| ステップ | A1 ログインする<br>A2 リポジトリを許可する<br>A3 ホーム画面に追加する<br>A4 アカウントを管理する | B1 起動する<br>B2 リポジトリを選ぶ<br>B3 内容を書く<br>B4 送信する<br>B5 結果を受け取る | C1 プリセットを作る<br>C2 ホーム画面・長押しメニューに置く | D1 作成した Issue を開く<br>D2 送信キューを確認する<br>D3 Issue を管理する |

> 活動の重み付けはミッションの優先順位（1. 起票の速さ・確実さ 2. セットアップの容易さ 3. 起票の表現力）に従う。**B が常に最優先**、次に A、C は B を速くするための投資、D は最小限から始める。

## 3. ストーリー詳細

書式:「〜として、〜したい、なぜなら〜」。ID は `{ステップ}-{連番}`。SP は [ミッションの工程別標準値](../project-mission.md) に基づく見積もり（合計・内訳は [04-milestones.md](04-milestones.md)）。

### 活動 A: セットアップする

| ID | MS | ストーリー | 補足・根拠 |
|----|----|-----------|-----------|
| A1-1 | M1 | 開発者として、PAT を発行せず GitHub アカウントのログインだけで使い始めたい。なぜなら PAT の取得・管理が既存の自作解（HTTP Shortcuts 等）の最大の摩擦だから | GitHub App + フルページリダイレクト + state/PKCE（[認証 §1, §3–4](../research/2026-07-10-auth-architecture.md)、[市場 §2](../research/2026-07-10-market-competitors.md)） |
| A1-2 | M1 | 開発者として、一度ログインしたら再ログインなしで使い続けたい。なぜなら起票のたびに認証が挟まると「起票 10 秒以内」が成立しないから | user token 8 時間 + リフレッシュトークン 6 ヶ月（単回使用・直列化必須。[認証 §2](../research/2026-07-10-auth-architecture.md)） |
| A2-1 | M1 | 開発者として、起票したいリポジトリへの App インストールを迷わず完了したい。なぜなら GitHub App は「インストール済み ∩ 自分がアクセス可能」なリポジトリにしか起票できず、ここで詰まると初回セットアップ 5 分以内を守れないから | オンボーディングで App インストールへ誘導する UX で制約を吸収（[認証 §1, §7](../research/2026-07-10-auth-architecture.md)） |
| A3-1 | M1 | 開発者として、アプリをホーム画面に追加したい。なぜならブラウザを開いて URL を辿る時間で「思いつき」は消えるから | PWA installable（manifest + SW + HTTPS）で WebAPK 化・リッチインストール UI（[UX §3](../research/2026-07-10-mobile-ux-pwa.md)） |
| A3-2 | M4 | 開発者でない利用者も含めて、Play ストアから普通のアプリとしてインストールしたい。なぜなら一般公開時、手動の PWA インストールより発見性と信頼感が高いから | TWA（Bubblewrap）。ライブサイト参照のためストア再リリース不要（[UX §6](../research/2026-07-10-mobile-ux-pwa.md)） |
| A4-1 | M1 | 開発者として、共有端末などで使い終えたらログアウトしてセッションを確実に無効化したい。なぜなら自分の GitHub へ起票できる経路を端末に残したくないから | FR-2。サーバー側セッション無効化（`POST /auth/logout`・[00-requirements.md §6.3](00-requirements.md)） |
| A4-2 | M1 | 開発者として、ログイン前に利用規約・プライバシーポリシーで収集データと免責を確認したい。なぜなら信頼できるか分からないサービスに GitHub の権限を渡す判断材料が要るから | FR-11・PR-1/PR-2。ログイン導線から参照できること（[00-requirements.md §8](00-requirements.md)） |
| A4-3 | M1 | 開発者として、使うのをやめるときはアカウント削除で本アプリ側のデータを完全に消したい。なぜなら暗号化されていてもトークンや設定が残り続けるのは望まないから | FR-12・PR-3。全テーブルの該当ユーザー行削除 + GitHub 側連携解除の案内（[00-requirements.md §6.2](00-requirements.md)） |
| A4-4 | M1 | 開発者として、他者のスパム起票で GitHub App が制裁・停止され自分まで使えなくなる事態を避けたい。なぜなら App 停止は全ユーザーに影響するから | PR-4/PR-6/PR-8。アプリ側レート制限・GitHub App の公開設定整備・匿名 API 利用経路を持たない（[00-requirements.md §8](00-requirements.md)） |

### 活動 B: 思いつきを起票する

| ID | MS | ストーリー | 補足・根拠 |
|----|----|-----------|-----------|
| B1-1 | M1 | 開発者として、起動後 1 秒以内に入力できる状態になってほしい。なぜなら「タスク管理は、アプリを起動するのすら面倒になったら終わる」から | app shell precache でサブ秒再訪（SW 起動コスト約 250ms）。Twitter Lite は compose 摩擦低減で投稿 75% 増（[UX §4](../research/2026-07-10-mobile-ux-pwa.md)、[市場 §3](../research/2026-07-10-market-competitors.md)） |
| B1-2 | M2 | 開発者として、ショートカットから起動したときはリポジトリ・ラベル・タイトル雛形が選択済みであってほしい。なぜなら選択をスキップできて初めて「3 タップ以内」の KPI が成立するから | `/new?repo=&labels=&title=` の URL パラメータ起動（[UX §3, §7](../research/2026-07-10-mobile-ux-pwa.md)） |
| B1-3 | M3 | 開発者として、起動直後の 1 タップでキーボードが開き即入力したい。なぜなら Android ではコールドローンチ時のキーボード自動表示が不可能で、「タップ → 同期 focus」が到達可能な最短だから | ボトムシート UI（vaul 等）+ `interactive-widget=resizes-content`（[UX §2](../research/2026-07-10-mobile-ux-pwa.md)） |
| B2-1 | M1 | 開発者として、リポジトリ名を検索して起票先を選びたい。なぜならリポジトリが増えても一覧スクロールでは 10 秒以内に収まらないから | App インストール済みリポジトリの取得 + インクリメンタル検索 |
| B2-2 | M1 | 開発者として、最近使ったリポジトリが先頭に出てほしい。なぜなら日々の起票先はほぼ数個に固定されるから | ローカル履歴（端末内保存）で十分。競合 QuickIssue も同機能（[市場 §1](../research/2026-07-10-market-competitors.md)） |
| B3-1 | M1 | 開発者として、タイトルだけで送信でき、必要なら本文も書きたい。なぜなら「タイトルのみ 5 秒以内」が KPI で、詳細は後から PC で書けばよいから | 起票フローに 1 タップでも追加する機能は既定でオフ（[ミッション判断基準](../project-mission.md)） |
| B3-2 | M2 | 開発者として、起票時にラベルを付けたい。なぜなら分類は起票の瞬間が最も低コストで、iOS 競合 QuickIssue はラベル非対応（要望あり）という差別化点だから | push アクセスがないと labels は silently dropped → B5-3 とセット（[認証 §6](../research/2026-07-10-auth-architecture.md)、[市場 §1](../research/2026-07-10-market-competitors.md)） |
| B3-3 | M3 | 開発者として、`#repo` `@label` のスマート入力でキーボードから手を離さず指定したい。なぜなら Todoist のクイック追加級の体験がミッションであり、ピッカー往復はタップ数を増やすから | トークンのインライン認識・ハイライト・タップ解除。純アプリロジックで PWA でも再現可能（[UX §1–2](../research/2026-07-10-mobile-ux-pwa.md)） |
| B3-4 | M2 | 開発者として、他アプリで見ている URL やテキストを共有シートから本文に取り込んで起票したい。なぜなら「思いつき」の多くは閲覧中のコンテンツ起点だから | Web Share Target（GET）。Android は共有 URL が `text` に入るため抽出処理が必要（[UX §3](../research/2026-07-10-mobile-ux-pwa.md)） |
| B4-1 | M1 | 開発者として、送信は 1 タップで完了してほしい。なぜなら確認ダイアログ 1 枚で「3 タップ以内」が崩れるから | Worker 経由の `POST /repos/{owner}/{repo}/issues`（[認証 §6](../research/2026-07-10-auth-architecture.md)） |
| B4-2 | M3 | 開発者として、電波が悪くても送信がキューされ、回復時に自動再送されてほしい。なぜなら「GitHub Issues に唯一欠けているのはオフライン対応」（Simon Willison）で、モバイルキャプチャ最大の穴だから | Workbox Background Sync（約 24h 保持・ネットワーク失敗のみ再送。4xx/5xx は UI 処理）（[UX §5](../research/2026-07-10-mobile-ux-pwa.md)、[市場 §3](../research/2026-07-10-market-competitors.md)） |
| B4-3 | M1 | 開発者として、送信ボタンの連打やタイムアウト再送で同じ Issue が二重に作られないでほしい。なぜなら GitHub API に冪等性キーがなく、タイムアウト再送は重複起票リスクだから | 基本の二重送信防止: 送信中の再タップ抑止 + サーバー側の直近送信内容（issue_log）照合（FR-24・[認証 §6](../research/2026-07-10-auth-architecture.md)） |
| B4-4 | M3 | 開発者として、オフラインキューからの自動再送でも同じ Issue が二重に作られないでほしい。なぜなら回復時の一括再送は重複起票の再発点になるから | B4-2 のキュー再送を B4-3 の issue_log 照合と統合する（[00-requirements.md](00-requirements.md) OQ-8 の検討結果を反映） |
| B5-1 | M1 | 開発者として、成功 / 失敗が明確に分かり、失敗しても入力内容が下書きとして残ってほしい。なぜなら成功率 99%・「無言で消失させない」が品質ゲートだから | 下書き保全はドメイン品質ゲート（[ミッション](../project-mission.md)）。楽観的 UI は M3 で強化 |
| B5-2 | M1 | 開発者として、認証切れ・権限不足・レート制限を区別できる言葉で知りたい。なぜなら原因が分からない失敗は再試行の判断ができず放棄につながるから | 422 は spam 判定を含むため盲目リトライ禁止・二次制限 80 req/min（[認証 §6](../research/2026-07-10-auth-architecture.md)） |
| B5-3 | M2 | 開発者として、push 権限のないリポジトリではラベルが反映されないことを送信前に知りたい。なぜなら API は 201 を返しつつラベルを silently drop し、静かな期待外れになるから | 権限に応じた UI 表示制御（[認証 §6](../research/2026-07-10-auth-architecture.md)） |

### 活動 C: ショートカットを整える

| ID | MS | ストーリー | 補足・根拠 |
|----|----|-----------|-----------|
| C1-1 | M2 | 開発者として、リポジトリ・ラベル・タイトル雛形を選ぶだけでプリセット起動 URL を作りたい。なぜなら手で URL パラメータを組み立てるのは開発者でも面倒で、間違えると起票が失敗するから | アプリ内ヘルパーで生成 → C2-2 の配置ガイドへ接続 |
| C2-1 | M2 | 開発者として、アイコン長押しメニューからよく使うプリセットを直接開きたい。なぜなら定番の起票先には検索すら省きたいから | manifest `shortcuts`（Android Chrome 最大 3 個・WebAPK 更新は約 24h 周期）（[UX §3](../research/2026-07-10-mobile-ux-pwa.md)） |
| C2-2 | M2 | 開発者として、4 個目以降のプリセットもホーム画面に置く手順を案内してほしい。なぜなら Chrome の「ショートカットを作成」は自力では気づきにくく、URL パラメータでショートカットを無限に作れるのが iOS 競合にない優位性だから | ブラウザバッジ付きの二級市民。standalone で開くかは未確認 → 要実機検証（[UX §3](../research/2026-07-10-mobile-ux-pwa.md)、[市場 §5](../research/2026-07-10-market-competitors.md)） |
| C2-3 | M4 | 開発者として、Play ストア版でも長押しショートカットをネイティブ品質で使いたい。なぜなら TWA 化で PWA の導線が失われては本末転倒だから | Bubblewrap は manifest `shortcuts` をネイティブ App Shortcuts に自動変換（[UX §6](../research/2026-07-10-mobile-ux-pwa.md)） |

### 活動 D: 起票後を確認する

| ID | MS | ストーリー | 補足・根拠 |
|----|----|-----------|-----------|
| D1-1 | M1 | 開発者として、起票成功後に作成された Issue を 1 タップで開きたい。なぜなら続きの詳細を書く・番号を控えるなど、直後に GitHub 側で作業したいことがあるから | 成功表示に Issue URL を含める（API レスポンスの `html_url`） |

| D3-2 | 将来 | 開発者として、起票時に自分や相棒をアサインしたい。なぜならチーム利用では担当が決まって初めてタスクとして動き出すから | `assignees` 配列を使用（API バージョン `2026-03-10` で単数 `assignee` 廃止）（[認証 §6](../research/2026-07-10-auth-architecture.md)） |
| D3-3 | 将来 | 開発者として、リポジトリの Issue テンプレートを使って起票したい。なぜなら公開リポジトリへの報告ではテンプレート準拠が求められるから | テンプレート取得・フォーム化は表現力（優先度 3）の範疇 |

## 4. KPI との対応

> KPI の正本は [`../project-mission.md`](../project-mission.md)（本表は対応関係の整理）。

| KPI | 主に担うストーリー | 補足 |
|-----|------------------|------|
| 起票 10 秒以内（タイトルのみ 5 秒以内） | B1-1（サブ秒起動）・B3-1（タイトルのみ送信）・B1-3（起動即入力）・B3-3（スマート入力） | M1 で 10 秒以内を成立させ、M2–M3 で短縮していく |

## 5. マップの運用

- ストーリーは GitHub Issues に `sp:` ラベル付きで起票し、本マップの ID（例: `B3-2`）を Issue 本文から参照してトレーサビリティを保つ（[session-sprint-rules.md](../rules/session-sprint-rules.md) の SP 付与経路に従う）
- スライスの入れ替え（ストーリーの前倒し・後送り）は本ファイルと [04-milestones.md](04-milestones.md) を同一 PR で更新する
- 実機検証で前提が崩れた場合（例: C2-2 のホーム画面ショートカットが standalone で開かない）は、リサーチ更新 → マップ更新の順で反映する
