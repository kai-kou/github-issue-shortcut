# マイルストーン計画 — GitHub Issue Shortcut

> 作成日: 2026-07-10。ストーリー ID（A1-1 等）は [03-user-story-map.md](03-user-story-map.md) を参照。
> SP は [プロジェクトミッションの工程別標準値](../project-mission.md)（1/2/3/5/8 + Dynamic 補正）による見積もりで、実測 4 週分で較正する。
> 開発は 1 セッション = 1 スプリントの自律運用（[session-sprint-rules.md](../rules/session-sprint-rules.md)）。タスクは GitHub Issues + `sp:` ラベルで管理し、各マイルストーンは GitHub Milestone に対応させる。

## M0 プロジェクト基盤

**ゴール**: コードを書き始める前に「push すればテストが走り、通れば本番へデプロイされる」状態と、認証に必要な GitHub App を整える。

ユーザーストーリーを持たない技術基盤（enabler）スライス。構成は [Cloudflare スタックリサーチ §7](../research/2026-07-10-cloudflare-stack.md) の推奨に従う。

本番 URL は workers.dev サブドメイン（名称は M0 で確定）で運用する。**カスタムドメインは現時点で取得しない**（[00-requirements.md](00-requirements.md) OQ-7 決定済み・2026-07-10）。GitHub App のコールバック URL / Setup URL は workers.dev の本番 URL で設定する。

### タスクと SP（合計 12 SP）

| タスク | 内容 | SP |
|--------|------|----|
| リポジトリ雛形 | Vite + React SPA + Hono API の単一 Worker 構成（`@cloudflare/vite-plugin`・wrangler v4・`wrangler.jsonc`・TypeScript・`@cloudflare/vitest-pool-workers`） | `sp:5` |
| CI パイプライン | GitHub Actions で test → deploy（`wrangler-action@v3`）。PR にはプレビュー環境 | `sp:3` |
| Cloudflare 環境 | 本番 / プレビューの Worker・D1・Secrets の設定 | `sp:2` |
| GitHub App 登録 | permissions: Issues=write のみで登録・client id/secret を Secrets へ・コールバック URL 設定 | `sp:2` |

### Done 判定（検証可能な条件）

- [ ] `main` への push で CI が test → deploy を実行し、テスト失敗時はデプロイされない
- [ ] 本番 URL で「Hello World」相当の SPA と Hono API（`/api/health`）が応答する
- [ ] PR 作成でプレビュー URL が得られる
- [ ] GitHub App が登録済みで、client id / client secret / トークン暗号鍵が Worker Secrets に設定されている
- [ ] `docs/` のアーキテクチャ前提（単一 Worker・D1）どおりの構成でローカル開発（`wrangler dev` / Vite）が動く

### リスク

| リスク | 影響 | 手当て |
|--------|------|--------|
| Cloudflare アカウント・GitHub App 登録に人間の操作（ダッシュボード・秘密情報の投入）が必要 | 自律スプリントが `status:waiting-user` で停止 | 必要な手順を 1 つの Issue にまとめて早期にユーザーへ依頼する |
| Vitest 4 系必須（vitest-pool-workers v0.13.0+）などツールチェーンのバージョン相性 | CI 構築の手戻り | リサーチ済みバージョン（[スタック §4](../research/2026-07-10-cloudflare-stack.md)）を初手で pin する |

## M1 MVP

**ゴール**: GitHub ログイン → リポジトリ選択（検索・最近使用）→ タイトル + 本文入力 → Issue 作成 → 成功 / 失敗表示（下書き保全）の一連が実機 Android で E2E 動作し、PWA としてホーム画面から起動できる。歩くスケルトンであり **この時点から開発者本人が毎日使う**。

| ストーリー | 内容 | SP |
|-----------|------|----|
| A1-1 GitHub ログイン | GitHub App OAuth（リダイレクト + state + PKCE）・セッション（D1・ハッシュ化 ID）・トークン AES-256-GCM 暗号化保管 | `sp:8`（新アーキテクチャ） |
| A1-2 ログイン維持 | リフレッシュトークンの単回使用ローテーションをユーザー単位で直列化 | `sp:3` |

| A3-1 ホーム画面追加（PWA 化） | manifest + Service Worker（app shell precache・`/auth/*` はナビゲーションフォールバック除外）・installable 要件充足。B1-1 サブ秒起動を含む | `sp:5` |
| B2-1 / B2-2 リポジトリ選択 | インストール済みリポジトリ取得 API + 検索 UI + 最近使用のローカル履歴 | `sp:5` |

### Done 判定（検証可能な条件）

- [ ] 実機 Android（Chrome）で、ログイン → リポジトリ検索 / 選択 → タイトル入力 → 送信 → GitHub 上に Issue 反映、を E2E で確認できる（品質ゲート: これなしで main にマージしない）
- [ ] ホーム画面追加で standalone（WebAPK）起動し、Lighthouse の PWA installable 要件に合格する
- [ ] standalone 起動からタイトルのみ起票が **10 秒以内** で完了する（実機計測）
- [ ] 機内モードで送信 → 失敗表示 + 入力内容が下書きとして残り、復元できる
- [ ] 認証切れ・権限不足・レート制限・Issues 無効がそれぞれ識別可能な文言で表示される
- [ ] 新規ユーザーがログイン → App インストール → 初起票を **5 分以内** に完了できる（実機で手順確認）
- [ ] standalone PWA での OAuth リダイレクト（CCT 経由 → scope 内コールバックで復帰）が実機で動作する

### リスク

| リスク | 影響 | 手当て |
|--------|------|--------|
| standalone PWA での OAuth はポップアップ不可・SW がコールバックを壊す既知問題（[認証 §4](../research/2026-07-10-auth-architecture.md)） | ログイン不能・M1 最大の技術リスク | フルページリダイレクト一択・pre-auth 値は Cookie・`/auth/*` を SW から除外。最初のスプリントで実機スパイクする |
| リフレッシュトークンの並行リフレッシュ競合で失効（[認証 §2](../research/2026-07-10-auth-architecture.md)） | 突然のログアウト → 成功率 KPI 毀損 | ユーザー単位の直列化（D1 ロック or Durable Object）。KV は書き込み特性上使わない（[スタック §3](../research/2026-07-10-cloudflare-stack.md)） |
| 「App 未インストールのリポジトリに起票できない」制約（[認証 §1](../research/2026-07-10-auth-architecture.md)） | 初回体験で「リポジトリが出ない」（競合 Anywhere Issue の致命的レビューと同型） | A2-1 のオンボーディング誘導を MVP スコープに含める |
| キーボード自動表示不可（[UX §2](../research/2026-07-10-mobile-ux-pwa.md)） | 「起動即入力」の期待とのギャップ | M1 では通常フォームで許容し、M3 のタップ → 同期 focus 設計で最短化 |
| MVP スコープ肥大（54 SP は最大のマイルストーン） | ドッグフーディング開始の遅れ | スライス内でも B 系（起票コア）を先に縦串で通す。表現力系の追加要望はすべて M2 以降へ |

## M2 ショートカット & ラベル

**ゴール**: 価値提案の核「リポジトリ / ラベルはショートカットが覚えている」を実現し、ショートカット起動 → 起票 3 タップ以内の KPI を成立させる。

### 主要ストーリーと SP（合計 20 SP）

| ストーリー | 内容 | SP |
|-----------|------|----|
| B3-2 ラベル指定 UI | ラベル取得 + 複数選択 UI。起票フローを遅くしない配置（既定は畳む） | `sp:5` |
| B5-3 ラベル権限の事前表示 | push アクセスなしでは silently dropped → 送信前に非反映を警告 | `sp:2` |
| B1-2 URL パラメータ起動 | `/new?repo=&labels=&title=` で初期選択済み起動 | `sp:3` |
| B3-4 Web Share Target | 共有シート受け（GET）+ `text` フィールドからの URL 抽出 | `sp:3` |
| C2-1 manifest shortcuts | 長押しメニューに定番プリセット（最大 3 個・アイコン整備） | `sp:2` |
| C1-1 / C2-2 ショートカット作成ヘルパー | プリセット URL の生成 UI + ホーム画面配置の手順ガイド | `sp:5` |

### Done 判定（検証可能な条件）

- [ ] `/new?repo=owner/name&labels=bug&title=雛形` で起動すると各項目が初期選択済みで表示される（未ログイン時はログイン後に復元）
- [ ] 実機の WebAPK でアイコン長押しメニューにショートカットが 3 個表示され、タップでプリセット済み画面が開く
- [ ] ヘルパーで作成したショートカットから「起動 → 入力 → 送信」が **3 タップ以内** で完了する（実機計測）
- [ ] push 権限のないリポジトリではラベル UI が警告表示になり、silently dropped を利用者が事前に知れる
- [ ] Android 共有シートに本アプリが表示され、共有したテキスト / URL が本文にプレフィルされる
- [ ] ラベルなし起票の所要時間が M1 から悪化していない（起票フロー計測の回帰確認）

### リスク

| リスク | 影響 | 手当て |
|--------|------|--------|
| manifest shortcuts は最大 3 個・WebAPK 更新は約 24h 周期（[UX §3](../research/2026-07-10-mobile-ux-pwa.md)） | プリセット追加が即反映されない・数が足りない | 4 個目以降はヘルパー（C2-2）のホーム画面ショートカットで補完。検証時は `SHORTCUTS_DIFFER` 更新周期を織り込む |
| ホーム画面ショートカットが standalone で開くか公式未確認（[UX §3](../research/2026-07-10-mobile-ux-pwa.md) のリスク項目） | ブラウザタブ起動だと体験が劣化 | M2 冒頭に実機スパイク Issue を立て、結果次第でヘルパーの案内文言・導線を調整する |
| ラベル UI が起票フローを遅くする | コア KPI の毀損 | ミッションの判断基準どおり「1 タップ追加する機能は既定でオフ」。ラベルはプリセット（URL パラメータ）主導で使わせる |
| 共有 URL が `text` に入る Android 挙動・manifest 変更後の再インストール要件（[UX §3](../research/2026-07-10-mobile-ux-pwa.md)） | 共有起票が動かない報告 | text からの URL 抽出を仕様化・検証手順に再インストールを明記 |

## M3 クイック追加体験

**ゴール**: 入力体験を Todoist のクイック追加級に引き上げ、オフラインでも「送れば必ず届く」信頼性で成功率 99% を自動化まで含めて満たす。

> **ボトムシート UI（B1-3）の優先度（2026-07-10 決定）**: 実現難易度次第で優先度を上げる。リサーチ上は成熟ライブラリ（vaul）+ `interactive-widget` で難易度は低〜中（`sp:3`）の見込み。**M2 完了時に半日スパイクで難易度を実測し、容易なら M3 冒頭で最優先着手**（スパイク結果次第で M2 末尾への前倒しも可）。

### 主要ストーリーと SP（合計 21 SP）

| ストーリー | 内容 | SP |
|-----------|------|----|
| B1-3 ボトムシート + 起動即入力 | vaul 等のボトムシート・`interactive-widget=resizes-content`・初回タップで同期 focus | `sp:3` |
| B3-3 スマート入力 | `#repo` `@label` トークンのインライン認識・ハイライト・タップ解除 | `sp:5` |
| B4-2 オフラインキュー | Workbox Background Sync（失敗 POST の IndexedDB キュー + 自動再送）+ 楽観的 UI（`useOptimistic`） | `sp:8`（新アーキテクチャ） |
| B4-4 キュー再送と重複防止の整合 | B4-2 の自動再送を M1 の issue_log 照合（B4-3）と統合し、一括再送でも重複起票させない（OQ-8 の検討結果を反映） | `sp:2` |
| D2-1 送信キュー確認 | キュー / 下書き一覧・再送・編集・破棄（4xx/5xx の手動救済導線） | `sp:3` |

### Done 判定（検証可能な条件）

- [ ] 機内モードで起票 → キュー表示 → 回復後に自動送信され、GitHub に **1 件だけ** 反映される（重複なしを含めて実機確認）
- [ ] API エラー（4xx/5xx）はキュー自動再送ではなく D2-1 の一覧に残り、手動で再送・破棄できる
- [ ] `#repo` `@label` が入力中にインライン認識・ハイライトされ、タップで解除できる
- [ ] ボトムシート表示中にキーボードを開いても送信ボタンが隠れない（`interactive-widget` の実機確認）
- [ ] タイトルのみ起票が **5 秒以内** で完了する（実機計測・KPI 上限値の達成）

### リスク

| リスク | 影響 | 手当て |
|--------|------|--------|
| Background Sync はネットワーク失敗のみ再送・約 24h 保持（[UX §5](../research/2026-07-10-mobile-ux-pwa.md)） | 「送ったのに届いていない」事故 | D2-1 のキュー可視化を同一マイルストーンに含め、楽観的 UI には必ず救済導線を付ける |
| 冪等性キーなし + 再送で重複起票（[認証 §6](../research/2026-07-10-auth-architecture.md)） | ゴミ Issue 量産・信頼喪失 | 基本の二重送信防止（B4-3）は M1 で導入済み。M3 では B4-4 で B4-2 のキュー再送を issue_log 照合と統合してから自動再送を有効化する |
| スマート入力のスコープ肥大（日付・優先度など Todoist 全機能の再現欲） | スプリント超過 | トークンは `#repo` `@label` の 2 種に限定。それ以外はバックログへ（YAGNI） |
| Issue 連続作成の二次レート制限（80 req/min・[認証 §6](../research/2026-07-10-auth-architecture.md)） | キュー一括再送時の失敗 | 再送は直列 + 間隔を空ける実装にする |

## M4 配布

**ゴール**: TWA 化して Play ストアで配布し、「Android×PWA の空白地帯」を一般ユーザーにも届く形にする。コードベースは PWA 1 つのまま（ライブサイト参照でストア再リリース不要）。

> **実施判断は保留（2026-07-10 決定）**: アプリ化（ストア配布）の優先度は低。M3 完了後もそのまま着手せず、改めてユーザーの判断を仰ぐ。実施判断時には Digital Asset Links 用のドメイン要否（OQ-7: 現在は workers.dev 運用）と Play 開発者登録（$25・人間作業）を合わせて再検討する。

### 主要ストーリーと SP（合計 13 SP）

| ストーリー | 内容 | SP |
|-----------|------|----|
| A3-2 TWA 化 | Bubblewrap でラップ・Digital Asset Links（assetlinks.json）・署名鍵管理 | `sp:8`（新アーキテクチャ） |
| Play ストア配布 | ストア掲載情報・スクリーンショット・内部テスト → 公開トラック | `sp:3` |
| C2-3 ネイティブ App Shortcuts | `twa-manifest.json` の shortcuts で長押しメニューをネイティブ化 | `sp:2` |

### Done 判定（検証可能な条件）

- [ ] Play ストア（少なくとも内部テストトラック）からインストールでき、URL バーなしのフルスクリーンで起動する（assetlinks 検証合格）
- [ ] TWA 版で M1–M3 の E2E 起票フロー（ログイン → 起票 → 反映）が回帰なしで動作する
- [ ] アイコン長押しでネイティブ App Shortcuts が表示され、プリセット起動が動作する
- [ ] Web 側のデプロイのみで機能更新が TWA 版に反映される（ストア再リリース不要の確認）

### リスク

| リスク | 影響 | 手当て |
|--------|------|--------|
| Play ストアのポリシー審査・開発者アカウント・署名鍵はユーザー（人間）の関与が必須 | 自律スプリントで完結しない | M4 開始時に必要な人間作業を 1 Issue に集約して先行依頼する |
