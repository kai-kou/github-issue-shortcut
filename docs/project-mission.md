# プロジェクトミッション（CP-5 の実体）

> このファイルはプロジェクトごとに書き換える。`core-principles.md` の CP-5 は
> 本ファイルを参照する形で汎用化されている。判断に迷ったら本ファイルのミッションに
> 最も貢献する選択をする。

## ミッションステートメント

「思いついた瞬間」を逃さず、Android スマホのホーム画面から数秒で特定の GitHub リポジトリに Issue を起票できる、最速・最短の起票体験を提供する。PAT 管理の手間なし（GitHub 認証）・ショートカット起動（リポジトリ / ラベル初期選択）を武器に、Todoist のクイック追加級の体験を GitHub Issues にもたらす。

## KPI / 成功指標

| 指標 | 目標 | 計測方法 |
|------|------|---------|
| 起票所要時間（起動 → Issue 作成完了） | 10 秒以内（タイトルのみなら 5 秒以内） | 実機計測 + クライアント計測イベント |
| 起票までのタップ数（ショートカット起動時） | 3 タップ以内（起動 → 入力 → 送信） | UX フロー定義で担保・実機確認 |
| 起票成功率（送信 → GitHub 反映） | 99% 以上（失敗時は入力内容を失わない） | エラーレート計測 |
| 初回セットアップ完了率（ログイン → 初起票） | 5 分以内に完了できる | オンボーディングフロー実機確認 |

## ミッションに基づく判断基準

| 判断場面 | ミッション指向の選択 |
|---------|-------------------|
| 機能追加 vs 起票速度 | 起票フローに 1 タップでも追加する機能は既定でオフ・後回し |
| リッチな UI vs 起動速度 | 起動速度優先。バンドルサイズ・初期ロードを常に監視 |
| 失敗時の挙動 | 入力内容の保全（下書き保持）を最優先。無言で消失させない |
| 汎用性 vs 単純さ | まず「自分が毎日使える」を満たす最小構成。汎用化は後（YAGNI） |

## 優先順位付け（迷ったとき）

1. 起票の速さ・確実さ（コア体験）
2. セットアップの容易さ（GitHub 認証・ショートカット作成）
3. 起票の表現力（ラベル・詳細・テンプレート）
4. ドキュメント・運用改善

## ドメイン固有の品質ゲート

- 起票フロー（ログイン → 起票 → GitHub 反映）の E2E 動作確認なしで main へマージしない
- GitHub API のエラー（認証切れ・権限不足・レート制限）をユーザーに分かる形で表示できない状態でリリースしない
- 送信失敗時に入力内容が失われる状態でリリースしない
- Lighthouse PWA installable 要件（manifest・Service Worker・HTTPS）を満たさない状態で「ホーム画面追加」機能を謳わない

## SP 工程別標準値（session-sprint-rules.md §3.2 の実体・#45）

> `session-sprint-rules.md` §3.1 のベーススケール + §3.1.5 Dynamic 補正を、本プロジェクトの工程に当てはめた **暫定値**。
> **実測（`content/analytics/sprint/` の tokens/sp・分/sp 中央値）が 4 週分貯まったら @owner が較正する** （§6）。

| 工程（本プロジェクトの典型タスク） | ベース SP | 備考 |
|------------------------------|-----------|------|
| 設定・データ更新・1 行修正 | `sp:1` | 機械的変更 |
| 単一コンポーネント / ドキュメントの軽微改修 | `sp:2` | 単一ファイル |
| 単一機能の実装（フォーム項目追加・API ハンドラ 1 本） | `sp:3` | 標準タスク |
| 複数ファイル横断の機能実装（認証フロー・PWA 化・ショートカット機構） | `sp:5` | フロント + API 横断 |
| 新アーキテクチャ導入（OAuth 基盤・オフラインキュー・TWA 化） | `sp:8` | 分割を検討（§3.1） |

**Dynamic 補正（横断適用・+1〜2 SP）**: 上記ベースに対し、要リサーチ・仕様未確定・新規領域・clarification 見込みが高い場合に加算する。SP は離散値（1/2/3/5/8）なので、加算結果は最も近い上位スケールに丸める。

- **上限**: ベース + Dynamic が `sp:8` を超える場合はキャップせず、`session-sprint-rules.md` §3.1/§3.1.5 に従いタスクを分割する（8 超のラベルは付けない）。

> **較正の約束（gaming 防止・§7）**: 本表は実測（tokens/sp）に基づき更新するが、velocity（done_sp）や tokens/sp 自体を目標値・ノルマにはしない。あくまで「見積もり精度を上げる」ためだけに使う観測値。
