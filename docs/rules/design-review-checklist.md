# デザインレビュー・チェックリスト（フロントエンド変更 PR 用）

> **SSOT**: 基準の根拠は `docs/design/design-guidelines.md`（D-1〜D-10・数値基準表）。
> 本シートは `src/`・`index.html`・manifest（`vite.config.ts`）に差分がある PR のレビュー観点を
> 1 枚ものに集約したもの。機械チェック済みの項目は再目視不要（Warning の妥当性確認のみ）。
> 運用は `docs/rules/self-review-checklist.md` と同じ（機械 = ツールが検出 / 目視 = レビューで確認）。

## 使い方

```
1. python3 tools/check_design_rules.py     # 静的チェック（Warning 一覧を確認）
2. npm run e2e                              # design-guidelines.spec.ts 含む E2E
3. 下表の「フロー」「目視」項目を差分に該当する範囲で確認
4. Warning を残す場合は PR 説明文「## 設計意図・既知の警告」に理由を明記
```

## 0. フロー影響（最重要・全フロントエンド PR 共通）

| ✓ | チェック項目 | 検証 |
|---|------------|------|
| ☐ | **起票フロー（起動 → 入力 → 送信）のタップ数・画面数・待ち時間が増えていない**（増える機能は既定オフか） | 目視（D-3） |
| ☐ | 必須入力が増えていない（タイトル以外はすべて省略可能のまま） | 目視（D-2） |
| ☐ | キャプチャ外の機能（閲覧・整理・編集）を持ち込んでいない | 目視（D-1） |
| ☐ | 新規依存（UI ライブラリ等）を追加していない。追加するなら別 Issue でバンドル影響を計測済みか | 目視（D-6/D-10） |
| ☐ | 初回セットアップ導線を変更した場合: ステップ数 3（ログイン → リポジトリ選択 → 初起票）を超えていないか・認可失敗の再試行導線があるか | 目視（§3.0） |

## 1. 入力・フォーム

| ✓ | チェック項目 | 検証 |
|---|------------|------|
| ☐ | フォームコントロールの font-size 16px 以上 | 機械（check_design_rules + E2E） |
| ☐ | 新規入力欄に enterkeyhint / inputmode / autocapitalize を検討したか | 機械（Warning）+ 目視 |
| ☐ | ラベルが入力欄の上に常時表示されている（placeholder のみでない） | 機械（Warning）+ 目視 |
| ☐ | 送信ボタンが画面下部（thumb zone）にあり、キーボード表示時も到達可能 | 目視（D-5） |

## 2. タップターゲット・視覚

| ✓ | チェック項目 | 検証 |
|---|------------|------|
| ☐ | インタラクティブ要素 24px 以上（最低）・44px 以上（推奨）・送信 48px | 機械（E2E boundingBox） |
| ☐ | 隣接ターゲットの間隔 8px 以上 | 目視 |
| ☐ | ライト/ダーク両モードで表示確認（コントラスト 4.5:1） | 機械（E2E smoke）+ 目視 |
| ☐ | アニメーション追加時: prefers-reduced-motion 対応 + 本当に必要か | 機械（Warning）+ 目視 |

## 3. 失敗時・データ保全

| ✓ | チェック項目 | 検証 |
|---|------------|------|
| ☐ | 失敗パス（401/403/429/404/422・オフライン）で入力が保持される | 機械（E2E issue-errors/draft）+ 目視 |
| ☐ | エラーメッセージに原因 + 次アクション + 下書き保存済みの明示がある | 目視（D-7） |
| ☐ | 下書き保存のトリガーを弱めていない（visibilitychange/pagehide） | 目視 |
| ☐ | 楽観的 UI を入れた場合、失敗時の明示エラー + 再送 UI がセットか | 目視（D-8） |
| ☐ | エラー表示時にフォーカスがエラー近傍へ移動し、隠されない位置に出るか（WCAG 2.4.3 / 2.4.11） | 目視（D-9・axe 導入後は #79 で自動化） |

## 4. PWA・起動速度

| ✓ | チェック項目 | 検証 |
|---|------------|------|
| ☐ | manifest 変更時: shortcuts 先頭が最優先アクション・3 件以内・アイコン 512/192 維持 | 目視 + E2E（pwa.spec） |
| ☐ | 初期ロードに重い処理・大きな依存を足していない（app shell の肥大化） | 目視（ビルドサイズ確認） |
| ☐ | viewport meta にズーム禁止（maximum-scale=1 等）が入っていない | 機械（check_design_rules） |

## 深刻度と対応

| 深刻度 | 対応 |
|--------|------|
| E2E 失敗 | 修正するまでマージしない（CI ブロッキング） |
| 静的チェック Warning | 修正する。残す場合は PR 説明文に理由を明記 |
| 目視項目の違反 | D-1〜D-3（フロー影響）は設計から見直す。それ以外は修正 or 理由明記 |

## 本シートの育て方

- AI レビュー・実機確認で同種のデザイン指摘が 2 回以上出たら、本シートに行を追加し、機械化可能なら `tools/check_design_rules.py` / `e2e/design-guidelines.spec.ts` に **同一 PR で** 追加する（`self-review-checklist.md` と同じフィードバックループ・L-094 desync 防止）
- 数値基準の変更は必ず SSOT（`docs/design/design-guidelines.md` §2）を先に更新する
