## §5. C カテゴリ自律処理ルール（ルール整備で解消済み/解消する事象・プロジェクト例）

> **実行モデルの明示**: project-sync スキルは `tools/sync_project.py` のような専用 Python 実装を持たず、**全て SKILL.md に規定され Claude が gh/mcp で直接実行する手順**（ID 採番・誤付与ラベル除去・auto-close スキャン・誤分類リセット等）で構成される。下表の「実装場所」はこの前提で記載する。下表の事象は **プロジェクト例**。

| 事象（プロジェクト例） | 自律処理 | 実装場所 |
|------|---------|---------|
| バックログ候補ラベルに waiting-user 誤付与 | ラベルを自動除去 | project-sync SKILL.md Step 2（**Claude が gh/mcp で実行するスキル手順**） |
| 公開完了済みの waiting-user Issue | 公開 URL 確認 → 自動クローズ | project-sync SKILL.md Step 2.5（**Claude が gh/mcp で実行するスキル手順**） |
| 週次/定例レポート系 Issue | Slack 通知済み + 7日経過 → 自動クローズ | project-sync SKILL.md Step 2.6（**Claude 実行**） |
| リサーチ・調査依頼が waiting-user 誤分類 | `status:waiting-claude` にリセット（担当スキルが自律実行） | project-sync SKILL.md Step 2.6（**Claude 実行**） |
| 環境変数未設定 | A 型チェックリスト → セッション env / secrets-broker 供給を確認（クラウドの `gh variable set` は 403・2026-07-02。未供給なら A-6 として設定名・手順付きで依頼） | problem-investigation-protocol.md A 型チェックリスト（**ドキュメント手順**） |

新たに C カテゴリの事象を発見したら、本表に追記し、対応する自律処理を project-sync（SKILL.md 手順）あるいは該当スキルに実装する。**実際にどのスキルの何ステップで実装したかを「実装場所」列に正確に記載すること**（L-094 desync 防止）。

## §6. D カテゴリ（外部要因）の自律最小化ルール（プロジェクト例）

外部課金・BAN・プラットフォーム制約は設定変更（A-6）のみユーザー作業だが、**影響を自律的に最小化** する。

下表は **プロジェクト例**。各プロジェクトが利用する外部 API・プラットフォームに読み替える。

| 外部要因（プロジェクト例） | 自律最小化策 | ステータス |
|---------|------------|-----------|
| 画像生成 API の請求ハード上限 | 枯渇前に Slack 予告。**実測検知時は `tools/api_depletion_alert.py` がハード上限到達を検知した瞬間に Slack 即通知（A-6）**。枯渇後は代替 provider の HTTP フォールバックで継続。代替エンジン降格は品質一貫性への影響評価が必要なため実装 Issue で慎重に検討 | 予告 + 実測即時通知 |
| 外部 API クレジット枯渇 | 予算トラッカーで枯渇前に予告（予測型）。**実測検知時は `tools/api_depletion_alert.py` が投稿/取得スクリプトのエラーパスでクレジット枯渇を検知した瞬間に Slack 即通知（A-6。推定残高が健全表示でも取りこぼさない）**。月初リセットを監視し復旧を自動確認 | 予告 + 実測即時通知 |
| GitHub Actions 課金超過 | Path フィルタで不要 CI を削減 + Lv3 フックで CI 前チェックを吸収し実行頻度を下げる | 実行頻度の継続的削減（プロジェクトで追跡） |

「枯渇したので止めます」で終わらせず、**フォールバック経路で可能な限り処理を継続** することが D カテゴリの自律運用の要件。
