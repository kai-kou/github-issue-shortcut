# 制約の強度ランク（ハーネスエスカレーション）

ハーネスエンジニアリングの原則に従い、制約を **「お願い」から「仕組み」へ** 段階的に強化する。
同じミスが 2 回以上繰り返された場合、上位レベルへの昇格を **必ず** 検討すること（セルフ改善ループと連動）。

| Lv | 種別 | 強制力 | 実装例 |
|----|------|--------|--------|
| 1 | **ドキュメント** | 低（AI が読む） | `CLAUDE.md`, `.claude/rules/*.md` |
| 2 | **AI セマンティック** | 中（スキルが検証） | `self-reviewer`, `fact-checker` スキル |
| 3 | **フック** | 高（物理的にブロック） | `post-tool-use-validate.sh`（台本 JSON バリデーション） |
| 4 | **CI / 構造テスト** | 最高（マージ不可） | 将来実装: GitHub Actions での品質ゲート |

## Lv3 フック詳細

- `content/scripts/*.json` の Write 後に自動実行
- JSON 構文・必須フィールド・方言ルール・数値や年号を含むのに `fact_check_flags` が空のセリフを検査する
- 新たなミスパターンを検出したら `post-tool-use-validate.sh` にチェックを追加して Lv3 を強化する

## 昇格手順

1. 同じミスが 2 回以上繰り返される → `retrospective` スキルで KPT を生成
2. Problem が Lv1（ドキュメント）で解消されない → Lv2（スキル）にセマンティックチェック追加
3. Lv2 でも解消されない → Lv3（フック）に物理バリデーション追加
4. `docs/rules/lessons/` 配下の該当カテゴリファイル（必要に応じて `docs/rules/lessons-core.md`）に Lesson として記録する
