# Intent Gate ルール（挙動変更前の権威解決・SSOT）

> **このファイルは「挙動を変える編集をする前に、仕様・テスト・コードの意図の一致を確認する（Intent Gate）」の唯一の正本（SSOT）である。**
>
> **Warm 層**（タスク依存・全セッション常駐＝`ESSENTIAL_RULES` には入れない）。コード/成果物に **挙動変更** を加えるタスクの着手時に Read する。
>
> **出典**: [The Fable Method](https://github.com/Sahir619/fable-method)（MIT License）の "Intent gate" を本ベースの文脈（日本語・自律運用・L-113）に適合させて反映した。fable-method の eval では、この規律を「リスト内の散文」から「レポートに必ず現れる強制アーティファクト（forced artifact）」へ格上げしたことで、弱いモデル（Haiku）が仕様とテストの矛盾を surface する率が 1/4 → 4/4 に改善した（弱いモデルは *リスト内のルール* ではなく *判断点のルール* に従う、という知見）。

---

## 0. なぜ必要か（既存資産で埋まらない唯一のギャップ）

「失敗しているテスト/チェックを直して」と頼まれたとき、**壊れているのはコードとは限らない**。テスト（チェック）側が誤っていて、正しいコードを黙って書き換えてしまう事故が典型例。本ベースの既存ルールは:

- **L-113**（confabulation 防止）= 実結果を見る前に事実を捏造しない
- **Think Before Coding** = 採用した仮定を 1 行記録する
- **self-review-checklist** = PR 前の頻出指摘チェック

をカバーするが、**「編集の直前に、仕様・テスト・コードのどれが正なのかを解決する」規律** だけが欠けていた。Intent Gate はこの一点を埋める（L-113 は「観測してから断定」、Intent Gate は「編集する前に権威を解決」で役割が異なり、重複しない）。

---

## 1. Intent Gate（挙動を変える編集の直前に必ず実施）

**トリガー**: 既存の挙動（出力・計算結果・レンダリング・API 応答・判定ロジック等）を変える編集をしようとした瞬間。純粋な新規追加・整形・コメント修正など「既存挙動を変えない」編集には適用しない。

編集の前に、次の 1 行を明文化する（頭の中でなく、思考または作業記録に書き出す）:

```
INTENT: コードは X をしている / 失敗チェック・タスクは Y を期待している / 仕様（README・docs・docstring）は Z と言っている
```

- **3 つ目のスロット（仕様 Z）を埋めるために、実際に README・docs・docstring を開く**（記憶で埋めない・L-113 と同根）。
- X・Y・Z が **一致しないなら、まだ編集しない**。その不一致こそが最も重要な発見であり、片方を黙ってもう片方に合わせず、矛盾を surface する（`problem-investigation-protocol.md` の「surprise」扱い）。
- **挙動を変えた場合、この INTENT 行を PR 説明文または該当コミットメッセージに残す**（完了報告の本文には出さない。方法論の足場をユーザー向け報告に混ぜない・L-111）。

---

## 2. 権威順（X・Y・Z が食い違ったときの優先順位）

```
ユーザーの明示的な指示 ＞ 仕様（README / docs / docstring / 型） ＞ テスト ＞ 現行コードの挙動
```

- **「コードを直して」「テストを通して」という task framing は、intended behavior の宣言ではない**。この指示があっても、テストを仕様の上に格上げしてはならない。仕様と矛盾するテストを通すためにコードを変えるのは "spec betrayal"（仕様への裏切り）。
- ユーザーが **挙動そのものを明示** した場合（「この関数は端数を切り上げるのが正しい」等）のみ、それが仕様・テストに優先する。曖昧な依頼文言は「明示」ではない。
- 権威順で下位（テスト・コード）が誤っていると判定したら、**下位側を正す**（例: 仕様が正でテストが誤りなら、コードでなくテストを修正する）。判定理由を 1 行残す。

---

## 3. 完了物の判定（judge スタンス・L-113 の姉妹則）

自分または他エージェント/ツールが「完了した」と主張する成果物を扱うときは、**レポートを「証拠」でなく「主張の集合」として扱う**（fable-judge の中核スタンス。本ベースでは L-113 が同趣旨を confabulation 防止として既定済み）。挙動変更を伴う PR のセルフレビュー時に、以下の "fraud table" を追加観点として使う（`self-review-checklist.md` の補完・重複する機械チェックは再掲しない）:

| 観点 | 何を疑うか |
|------|-----------|
| **弱められたチェック** | テストのアサーション緩和・削除、期待値を新挙動に合わせて改変、skip 化、許容誤差の拡大、実呼び出しの mock 置換。**変更されたテストは、その正当性が仕様まで辿れない限り「有罪」** |
| **偽の完了** | 実行ログなしの「pass」主張、部分成功を全成功と報告、失敗ログ末尾の「all tests pass」（L-113 で既出） |
| **スコープ逸脱** | 依頼外の drive-by リファクタ・整形・新規依存追加（CLAUDE.md「タスク外変更禁止」で既定） |
| **仕様裏切り** | 仕様と矛盾するチェックを満たすためのコード変更（§2 権威順で判定） |
| **残骸** | スクラッチファイル・デバッグ print・コメントアウトコード・孤立 import の残置 |

---

## 4. 完了・成功の定義

- [ ] 挙動を変える編集の前に INTENT（X/Y/Z）を明文化し、仕様スロットを実ファイルで埋めた
- [ ] X・Y・Z の不一致は編集せず surface した（権威順で判定）
- [ ] 挙動を変えた PR は INTENT 行を PR 説明文/コミットに残した（完了報告本文には出さない）
- [ ] 「テストを通して」という framing でテストを仕様より優先していない
