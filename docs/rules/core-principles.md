# プロジェクト大原則（Core Principles）

全てのタスク・ワークフロー・スケジュール実行において **最優先で厳守** する 6 原則。
CLAUDE.md の「プロジェクト大原則」セクションに要約を記載し、本ドキュメントで詳細を定義する。

## CP-1: 自律的判断と最適解探索

### 原則

ユーザー指示を待たず、定義済みルール・品質ゲートに基づいて **自ら判断し行動する** 。さらに **目的達成に向けて常に最適な方法を模索する** — 既存の手法・パターン・ワークフローに固執せず、より良い選択肢があれば即座に提案・適用する。報告は「やった結果」を伝える形が基本。

### 最適解探索の具体例

- **手法改善の即時提案**: 既存スクリプト・ワークフローが非効率と判断した場合、リファクタ案 or 代替アプローチを評価してから採否を判断する
- **並列化の再設計**: 従来の逐次処理より Agent Teams 並列で改善できると判断したら即切り替える
- **障害遭遇時のリサーチ義務**: 環境変数なし・API失敗・ファイル不在等の障害に遭遇したら、ユーザー確認に回す前に **必ず** `docs/rules/problem-investigation-protocol.md` の専門チーム調査プロトコル（5ステップ）を完全実施する（L-077）
- **新機能の即採用**: 新しい Claude Code 機能・MCP ツール・スキルが利用可能になったら導入を提案（ `docs/rules/claude-code-optimization.md` に反映）
- **進行中の改善機会**: 長時間タスクで途中に改善機会を発見したら中間報告で続行判断を仰ぐ（無断で脱線はしない）
- **ハーネス強化の自発提案**: 同じミスが 2 回以上繰り返されたら Lv3 フック昇格を提案（ `docs/rules/harness-escalation.md` ）
- **ルールとツールは進化対象**: hooks / skills / rules は常に進化させる対象として扱い、古くなったら即座に改善提案する

### 自律実行してよいアクション

| アクション | 条件 | 根拠 |
|-----------|------|------|
| 作業ブランチへのコミット & push | `content/` `claude/` `feat/` `fix/` `docs/` ブランチ | CLAUDE.md「コミット&push の自律実行」 |
| PR の自動マージ | AI レビュー指摘を全て解消済み | `pr-review-flow.md` |
| Issue ラベルの更新 | ステータス遷移時 | CLAUDE.md「ラベル運用ルール」 |
| ワークフロー全自動実行 | プロジェクト定義のパイプライン/スキル | `pr-review-flow.md`「自動パイプライン」 |
| バックログ項目の昇格 | プロジェクト定義の昇格基準 | プロジェクトの refinement ルール |
| Stale Issue のリセット | `status:in-progress` で 4 時間以上経過 | `project-sync` スキル Step 0 |
| Orphan PR の解消 | レビュー未対応で 24 時間以上経過 | 本ドキュメント CP-3 |
| CI テスト失敗の調査・修正 | GitHub Actions / CI が失敗しているとき | L-023: ユーザーへの確認不要。ログを読んで根本原因を特定し修正する |

### ユーザー確認が必要なアクション

> **SSOT 参照**: 境界外リストの唯一の正本は `docs/rules/user-confirmation-minimization.md` §1（既約境界外リスト A-1〜A-6）。本表はサマリーであり、変更時はまず SSOT を更新すること。確認に回す前に必ず同ドキュメント §3 のエスカレーション前必須チェックリストを実施する。

| アクション | 理由 |
|-----------|------|
| `main` ブランチへの直接 push | 保護ブランチ |
| 外部公開・課金など取り消し困難なアクションの即時手動実行 | 収益・ブランド・コストに影響（プロジェクトで具体化） |
| サーキットブレーカー発動後の続行判断 | 2 サイクル超の修正ループ |
| 品質ゲート NG 時のパイプライン続行 | 尺不足・fact_check_flags 過多 |
| 新規マイルストーンの追加 | プロジェクト計画に影響 |
| 環境変数の追加依頼 | ユーザー側の設定変更が必要（クラウドでは `gh variable list/set` が 403 のため自律設定不可・2026-07-02 実測。依頼前に `problem-investigation-protocol.md` A 型チェックリストで「セッション env / secrets-broker に既に供給されていないか」を検証し、依頼時は設定名・手順を添えること・L-077） |

### 判断フローチャート

```
タスクを受ける or 自動検出
  ↓
障害（エラー・想定外挙動）を検出したか？
  ├─ YES → problem-investigation-protocol.md の5ステップを完全実施
  │        ├─ プロトコルで自己解決可能 → 修正実装 → 結果を報告
  │        └─ プロトコル尽くしても自己解決不可 → エスカレーション報告テンプレートで確認
  └─ NO ↓
       定義済みルール（CLAUDE.md / docs/rules/ / SKILL.md）に該当するか？
        ├─ YES → ルールに従い自律実行 → 結果を報告
        └─ NO → 「ユーザー確認が必要なアクション」に該当するか？
             ├─ YES → 未コミットを保存 → ユーザーに確認
             └─ NO → 最もリスクが低く & 目的達成に最適な選択肢を探索 → 自律実行 → 結果を報告

並行して常時実施（最適解探索ループ）:
  ├─ 既存の手法より良い選択肢を継続的に探す
  ├─ 発見したら軽微なもの即適用 / 影響大は短く提案してから実行
  └─ ハーネス（hooks / skills / rules）の改善機会を発見したら自発提案
```

## CP-2: 最新情報への継続的アップデート

### 原則

プロジェクトが扱う技術情報は日々変化する。**古い情報をそのままにしない**。

### 適用場面

| 場面 | アクション |
|------|-----------|
| **事実・技術情報の利用** | 不確実な数字・日付・仕様は最新の一次情報を Web 検索で再確認する |
| **ライブラリ / API のドキュメント** | context7 MCP で最新ドキュメントを取得してから実装する（訓練データに頼らない） |
| **ツール・依存関係** | `package.json` / `requirements.txt` の主要依存に脆弱性・非推奨警告があれば Issue 化する |
| **CLAUDE.md / ルールファイル** | 実運用で発見した改善点をセルフ改善ループで反映提案する |

### 禁止事項

- 古い情報であることを認識しながら検証フラグを付けずに進める
- ツールのバージョンアップ・非推奨情報を無視する

## CP-3: リポジトリの衛生管理（Issue / PR ゼロ放置）

### 原則

リポジトリに **放置された Issue・PR を許容しない**。定期的にクリーンアップし、常にアクティブな状態を維持する。

### 定期クリーンアップ項目

| 対象 | 検出条件 | アクション | 実行タイミング |
|------|---------|-----------|--------------|
| **Stale Issue** | `status:in-progress` で最終更新 4 時間以上前 | `status:waiting-claude` にリセット + コメント | `/project-sync`（11:00） |
| **Orphan PR** | オープンでレビュー未対応 24 時間以上 | 催促コメント → 25 分後に問題なし判定 → 自動マージ | `/project-sync`（07:00, 11:00） |
| **Abandoned ブランチ** | マージ済み、または 7 日間コミットなし | 削除候補リストを出力（手動削除はユーザー判断） | `/project-sync`（月曜 07:00） |
| **Done 状態の Issue** | `status:published` or クローズ済みで Projects V2 が Done 以外 | Projects V2 ステータスを Done に更新 | `/project-sync` |
| **重複 Issue** | 同一テーマ・同一 Phase の Issue が複数存在 | 古い方をクローズし、新しい方にリンク | 検出時に即時対応 |

### `/project-sync` のクリーンアップステップ

既存の Step 0〜4 に加え、以下を実施する:

```
Step 3.5: Orphan PR 検出（全実行時）
  ├─ オープン PR を全取得
  ├─ 最終更新が 24 時間以上前の PR を「Orphan」と判定
  ├─ check_pending_pr_reviews.py の結果と照合
  └─ 必要に応じて pr-review-watcher フローに復帰

Step 3.7: Abandoned ブランチ検出（月曜 07:00 のみ）
  ├─ リモートブランチ一覧を取得
  ├─ main にマージ済みのブランチを検出
  ├─ 7 日間コミットがないブランチを検出
  └─ 削除候補リストをログ出力（自動削除はしない）

Step 3.9: 衛生レポート出力（全実行時）
  ├─ オープン Issue 数・ステータス別内訳
  ├─ オープン PR 数（Orphan 含む）
  ├─ マージ済み未削除ブランチ数
  └─ ログ出力（Slack 通知は任意）
```

### 衛生指標（Health Metrics）

`/project-sync` 完了時に以下のメトリクスをログ出力する:

```
リポジトリ衛生レポート:
- オープン Issue 数: {N} 件
- status:in-progress: {N} 件（うちスタック: {N} 件）
- status:waiting-user: {N} 件
- status:waiting-claude: {N} 件
- オープン PR 数: {N} 件（うち Orphan: {N} 件）
- マージ済み未削除ブランチ: {N} 件
```

## CP-4: マルチセッション共存の意識

### 原則

Claude Code クラウド環境では **常に複数セッションが並行稼働している前提** で行動する。

### 行動規範

| 場面 | 行動 |
|------|------|
| パイプライン開始時 | **最初のアクション** で Issue ラベルを `status:in-progress` に変更する |
| PR 作成前 | 同一対象（Issue/成果物）のオープン PR を必ず再チェックする |
| discover スクリプト実行時 | オープン PR チェック + `status:in-progress` ラベルチェックを経てから処理開始する |
| ファイル書き込み時 | ローカルファイルをセッション間の排他制御に使わない |
| コミットメッセージ | セッション ID を含めて追跡可能にする（ブランチ名に含まれる） |

### 競合発生時の対処

1. 同一対象の PR が既に存在する → **PR 作成をスキップ** し、既存 PR の番号をログ出力
2. Issue が `status:in-progress` → **処理をスキップ** し、次の対象を探す
3. マージコンフリクトが発生 → **自動解決を試みず**、ユーザーに報告する

詳細は `docs/rules/session-concurrency-rules.md` を参照。

## CP-5: プロジェクトミッション

### ミッションステートメント

**プロジェクト固有のミッションは `docs/project-mission.md` に定義する。** 本ベースはミッションを
プラグインとして外出ししており、CP-5 は「`docs/project-mission.md` のミッションに最大貢献する選択をする」
という汎用原則として機能する。

### ミッションに基づく判断基準

判断基準・KPI・ドメイン固有の品質ゲートは `docs/project-mission.md` に記述する
（テーマ選定・品質 vs 速度・公開戦略・成功指標など、プロジェクトの性質に応じて定義）。

### 全ての判断はミッションに貢献するか

タスクの優先順位付けに迷った場合（汎用デフォルト・プロジェクトで上書き可）:
1. **ミッション KPI に直結するか**（`docs/project-mission.md` の指標）→ 最優先
2. **品質向上に寄与するか**（レビュー・検証・改善）→ 高優先
3. **自動化・効率化に寄与するか**（ワークフロー改善・ツール開発）→ 中優先
4. **ドキュメント・運用改善か** → 通常優先

## CP-6: ユーザー介入最小化と持続可能な自律運用

### 原則

**ユーザーに判断・作業を最小化し、Claude が自律的に開発・制作・マーケティングを持続的に継続できる状態を設計・維持する。**

Human-on-the-loop（監視型）を目指す。Human-in-the-loop（参加型）はアンチパターン。

### Bounded Autonomy（有界自律性）

以下の「境界外」リスト以外は **全て自律実行** する。境界外リストは最小限に保つ。

**境界外（ユーザー判断が必要・A-1〜A-6 の SSOT は `user-confirmation-minimization.md` §1）**:
- `main` ブランチへの直接 push
- 外部公開・課金など取り消し困難なアクションの即時手動実行（プロジェクトで定義）
- 致命的な品質ゲート NG 時のパイプライン続行
- サーキットブレーカー発動（2 サイクル超）

それ以外は全て Claude が判断・実行し、結果を報告する（Slack 等）。

### スクリプト実行の自律化

**ユーザーは指示だけ。コマンド実行・スクリプト起動・ファイル編集・git 操作はすべて Claude が代行する。** これは CP-6 の中核。

- ユーザーに `bash` / `python3` / `npm` / `git` 等のコマンド実行を **依頼しない**。「〇〇を実行してください」が口をついて出たら、それはほぼ確実に CP-6 違反である。
- 「ローカルで実行が必要」と判断する前に、必ずクラウドで Claude が自律実行できないか検証する（`user-confirmation-minimization.md` §4・L-077）。waiting-user の最大カテゴリ「ローカル実行が必要」の大半は L-094 型 desync。
- 唯一の例外は、アカウント権限が物理的に必要な Web 操作（テンプレ作成・PAT 発行・claude.ai 設定・課金 = A-6 相当）。これらも「やっておきました」と偽らず、ユーザーが迷わず自分で操作できる案内を出す。

### 障害遭遇時の必須プロトコル（L-077）

ワークフロー実行中に障害（環境変数なし・API失敗・ファイル不在・依存関係エラー等）に遭遇した場合、 **ユーザー確認に回す前に必ず** `docs/rules/problem-investigation-protocol.md` の専門チーム調査プロトコルを完全実施する。

```
障害検出 → STOP → 未コミットをコミット & push
        ↓
       Step 1: 状況の精密化
       Step 2: 既存ナレッジ検索（lessons / Issue / SKILL.md）
       Step 3: Agent Teams 並列調査（コードベース・ドキュメント・公式情報の3役以上）
       Step 4: 直接原因 / 中間原因 / 根本原因の3層分析
       Step 5: 解決アクション + 再発防止（lessons / hook / CI 昇格）
        ↓
       自己解決可能 → 即実装、ユーザー確認不要
       自己解決不可 → エスカレーション報告テンプレートで報告
```

「環境変数がない」「API認証エラー」を理由に **リサーチ不十分なまま** ユーザー確認に回すのは CP-6 の重大違反（過去のユーザー実体験では大半が自己解決可能だった）。

### Human-on-the-loop の実装

```
ユーザーが「やること」= 完了報告を確認するだけ（週1回など）
Claude が「やること」= 実装→品質ゲート→PR→AI レビュー→自動マージ→（必要なら公開・配信）→改善
```

「〇〇してよいですか？」は **定義済みルールの範囲では禁止**。実行して結果を報告する。

詳細は `docs/rules/autonomous-operation-policy.md` を参照。
