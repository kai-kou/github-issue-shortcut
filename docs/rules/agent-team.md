# サブエージェント・エージェントチーム活用ルール

タスク対応時は積極的にサブエージェント（Agent tool）を活用し、並列実行で効率を最大化する。

## 基本方針

- 独立した作業は **並列サブエージェント** で同時実行する。逐次処理より並列処理を優先する
- 調査・リサーチ系タスクはリサーチ専任のサブエージェント（Agent tool の `subagent_type=Explore`）に委譲し、メインエージェントは結果の統合と判断に集中する
- 複雑な実装計画は設計・プランニング専任のサブエージェント（Agent tool の `subagent_type=Plan`）に先行して検討させ、その成果をもとに実装タスクを分割する

## 2つの協調モード（fan-out vs 議論型）と用語整理

本ドキュメントが扱う「エージェントチーム」は、歴史的に **役割分担型 fan-out**（各サブエージェントが独立評価 → メインが集計・議論なし）を指してきた。これは Claude Code 公式の **Agent Teams 機能**（`TeamCreate`/`SendMessage` による peer-to-peer・実体は別 Claude Code インスタンス）とは別物である点に注意する。

| モード | 実体 | エージェント間通信 | 用途 | 参照 |
|--------|------|------------------|------|------|
| **役割分担型 fan-out**（既定） | Agent tool の並列サブエージェント | なし（メインが集計） | 多角的な独立チェック・速い | 本ドキュメント |
| **議論型（敵対的相互レビュー）** | `claude -p` lead + サブエージェント + 共有ホワイトボード | あり（ホワイトボード経由で相互に読み反論） | 過剰指摘の相互排除・合意形成・根本原因特定 | `docs/rules/discussion-whiteboard-rules.md`（実装: `tools/run_discussion_review.py` + `tools/discussion_whiteboard.py`） |
| 公式 Agent Teams（参考） | 別 Claude Code インスタンス | あり（peer-to-peer メッセージ） | 対話セッションでのアドホック議論。headless 自動運用は lead/teammate 早期終了リスク | `docs/rules/agent-team.md`（本表）・公式 docs |

> **使い分け（SSOT は `agent-team-summary.md`「2 協調モードと振り分け」）**:
> - **ユーザーが明示的に「専門チームを組成して」「専門チームで」「チームで議論して」等と指示したら、議論型（`claude -p`）を既定とする**（飼い主決定）。fan-out で代替してよいのはコスト/速度優先の明示か軽微タスクのときだけ（理由を 1 行述べる）。
> - 内部の自動調査（`problem-investigation-protocol.md` Step 3 の障害調査）・定型チェックは fan-out が既定（軽量・速い）。
> - **過剰指摘や誤検知が問題になるレビュー**（画像/台本レビュー等）も議論型を選び、各エージェントに互いの指摘を批判検証させて critical を相互検証済みに絞る。
>
> 実機検証により `Workflow`/`TeamCreate`/`SendMessage` は **`claude -p` サブプロセス内でのみ利用可能**（ハーネス本体には非露出）であることが確定している（#2588）。

## Phase別の並列化パターン

> ⚠️ 以下の表は **出自プロジェクト（動画制作パイプライン）の実例**。自プロジェクトのワークフローの
> 各フェーズに読み替えること（「独立した作業を並列サブエージェントで同時実行する」原則が本体）。

| フェーズ / タイミング | 並列化の例 |
|-------------------|-----------|
| Phase 1（ネタ収集） | 複数ニュースソースの巡回を並列エージェントで同時実行 |
| Phase 2（リサーチ） | リサーチ結果のファクトチェックと網羅性チェックを並列実行 |
| Phase 3（台本制作） | **セクション単位で分割生成**（2〜3セクションずつ並列サブエージェント）→ メインが統合。品質チェックも並列 |
| Phase 4（素材制作） | 音声合成・画像生成・動画素材取得を並列エージェントで同時進行 |
| Phase 5（動画制作） | レンダリング後の自動レビュー項目（尺・同期・無音区間）を並列チェック |
| 各PhaseのPR作成後 | AIレビュー依頼（Gemini / Copilot）を並列で送信 |

## エージェントチーム構成の例

複数工程にまたがるタスクでは、以下のようにエージェントチームを編成する。

```
メインエージェント（オーケストレーション）
├── Explore エージェント（コードベース調査・ファイル探索）
├── 実装エージェント A（ファイル編集・コード生成）
├── 実装エージェント B（別ファイルの並行編集）
└── 検証エージェント（テスト実行・品質チェック）
```

## 使い分けガイドライン

| 状況 | 推奨アプローチ |
|------|--------------|
| 単一ファイルの軽微な修正 | サブエージェント不要。直接対応 |
| 複数ファイルの調査が必要 | `Explore` エージェントに委譲 |
| 独立した2つ以上の作業 | 並列サブエージェントで同時実行 |
| 大規模な実装タスク | `Explore` (必要に応じて) → `Plan` → 並列実装エージェント → 検証エージェント |
| ファイル探索 + 実装が必要 | Explore で調査 → 結果を受けて実装（逐次） |

メインセッションおよびサブエージェントで使用するモデルを、タスクの複雑さに応じて使い分ける。

### モデル一覧（2026-07-01 時点）

| モデル | モデル ID | コンテキスト | 料金（入力/出力 per 1M） | Thinking 方式 | 推奨 effort | 用途 |
|--------|----------|------------|----------------------|--------------|------------|------|
| **Opus 4.8** | `claude-opus-4-8` | 1M トークン | $5 / $25 | Adaptive Thinking | `xhigh`（既定は `high`） | **台本生成（Phase 3）**、複雑な設計判断、大規模リファクタリング。最も自律的・長期エージェント作業に最強（2026-05-28・v2.1.154 でデフォルト化） |
| Opus 4.7（legacy） | `claude-opus-4-7` | 1M トークン | $5 / $25 | Adaptive Thinking | `xhigh` | Opus 4.8 への移行期の代替。SWE-bench Pro 64.3%（2026-04-16 リリース） |
| Opus 4.6（legacy） | `claude-opus-4-6` | 1M トークン | $5 / $25 | Adaptive Thinking | `max` | さらに旧世代。新規利用は非推奨 |
| **Sonnet 5** | `claude-sonnet-5` | 1M トークン | $3 / $15 | Adaptive Thinking | `medium` | メインセッションのデフォルト、実装エージェント、PR作成・レビュー対応 |
| **Haiku 4.5** | `claude-haiku-4-5` | 200K トークン | $1 / $5 | Extended Thinking | — | Explore エージェント、ファイル探索、パターン検索、セルフレビューの個別チェック |

> **Opus 4.8（2026-05-28・Claude Code v2.1.154 でデフォルト化）**: 最も自律的で、長期エージェント作業・ナレッジワーク・メモリが SOTA。Opus 4.7 と **API リクエスト面の破壊的変更はなし**（thinking は adaptive のみ。`budget_tokens` / `temperature` / `top_p` / `top_k` を送信すると **HTTP 400 エラーで拒否** される＝これらのパラメータは使用不可）。**重要: デフォルト effort が `high`**（Opus 4.7 は `xhigh`）。Opus 4.7 → 4.8 へ切り替えると effort が自動的に `high` にリセットされる。公式推奨は「反射的に `xhigh` を選ばず `high` を基準に eval で per-route 調整」。コーディング/エージェント・台本生成など知能重視タスクは `xhigh` を明示指定する。
> **Opus 4.8 の挙動の癖（プロンプトで調整）**: ① 4.7 よりナレーション増・タスク完了時の「他にも〇〇しましょうか？」が増える（サイレント既定＋小決定は聞かず進める指示で抑制） ② 検索・サブエージェント・ファイルメモリ・カスタムツールに消極的（「いつ使うか」を明示すると改善） ③ 長期タスクは目標を冒頭で明示すると精度が上がる（Claude Code は `/goal` が有効）。
> **Haiku 4.5 は Extended Thinking に対応**（Claude 4 Haiku）。高速・低コストで複雑な推論も可能。モデル ID は `claude-haiku-4-5`（日付サフィックス不要）。

### Adaptive Thinking（適応的思考）

Opus 4.8 / Opus 4.7 / Opus 4.6 / Sonnet 5 は **Adaptive Thinking** を搭載。クエリの複雑さに応じて、内部で「どれだけ考えるか」を自動調整する。Opus 4.8 / 4.7 は **adaptive 固定**（固定 thinking budget は使えない・`CLAUDE_CODE_DISABLE_ADAPTIVE_THINKING` は無効）。

- 単純なタスク → 思考を最小化（速く・安く）
- 複雑なタスク → 深く考える（品質優先）
- **`/effort` コマンド** で明示的に制御可能（CLI のみ。ブラウザ版はステータスバーのドロップダウンで変更）。5段階（v2.1.105+）:
  - `/effort low` — 速度優先（定型チェック・軽量タスク）
  - `/effort medium` — トークン節約（API デフォルトより低負荷）
  - `/effort high` — 複雑なタスクに対応
  - `/effort xhigh` — **Opus 4.7 のデフォルト / Opus 4.8 では明示指定が必要**（4.8 既定は `high`）。コーディング・エージェント・台本生成に推奨
  - `/effort max` — 絶対最大・Opus 専用。correctness がコストより重要な時のみ
  - （Opus 4.8 セッション限定）`ultracode` — xhigh + Dynamic Workflows（`/workflows`）。大規模・多エージェントの動的オーケストレーション。`settings.json` には書けない

### Interleaved Thinking（ツール呼び出し間の思考）

Claude 4 モデル全般で対応。ツール結果を受け取った後に内部思考を挟めるため、**マルチステップのエージェント動作** で品質が向上する。

- multi-agent pipeline の精度向上に活用
- 特に `script-pipeline` / `audio-pipeline` で顕著な効果が期待できる

### コスト最適化の優先順位

| 戦略 | 削減効果 | 適用タイミング |
|------|---------|-------------|
| **Haiku を使う** | 60〜80% 削減（vs Opus） | 調査・検証・チェック系タスク |
| **Prompt Caching** | 最大 90% 削減 | 繰り返しリクエスト（キャラ設定・ルールの再注入） |
| **Batch API** | 50% 削減 | 非同期・大量処理（theme-discovery の大量分析） |
| **キャッシュ + バッチ + Haiku 組み合わせ** | 最大 95% 削減 | 可能な限り組み合わせる |

**メインセッション** :
- デフォルトは **Sonnet 5** を使用する
- 長文生成や複雑な設計判断が必要な場合は **Opus 4.8** に切り替える（旧: Opus 4.7）
- Opus 4.8 / Opus 4.7 / Sonnet 5 ともに **1M トークン** のコンテキストウィンドウを持つため、長時間パイプラインでも圧縮が発生しにくい（Claude Code では Max/Team/Enterprise プランで 1M に自動アップグレード）

**サブエージェント** （Agent tool の `model` パラメータで指定）:
- `subagent_type=Explore` → `model="haiku"` （コードベース調査・ファイル探索）
- `subagent_type=Plan` → `model="sonnet"` （実装計画・設計）
- `subagent_type=general-purpose` （実装） → `model="sonnet"`
- `subagent_type=general-purpose` （検証・チェック） → `model="haiku"`

**Haiku 推奨スキル**（`project-sync` / `retro-try-handler` / `theme-discovery` は SKILL.md frontmatter に `model: haiku` を設定済み。`sns-publisher` はフェーズ混在のため frontmatter ではなく本文セクションでモデル使い分けを定義）:

| スキル | Haiku 適用範囲 | 理由 |
|-------|--------------|------|
| `project-sync` | 全フェーズ（frontmatter に設定） | Issue/PR 状態の定型チェック・ラベル更新のみ |
| `retro-try-handler` | 検出・ルーティングフェーズ（frontmatter に設定） | Issue 検出・ルーティング判断が主。実装ステップは Sonnet へ切り替える |
| `theme-discovery` | スコアリングフェーズ（frontmatter に設定） | スコア計算・Issue 更新のみ。台本化判断フェーズは Sonnet を維持 |
| `sns-publisher` | 検出フェーズのみ（本文セクションで定義） | 配信対象動画の検出・YAML 読み込みのみ。記事生成フェーズは Sonnet を使用 |

**原則** :
- 迷ったら Sonnet を選ぶ。Opus は明確に必要な場合のみ
- 並列サブエージェントは可能な限り Haiku を使い、コストを抑える
- 品質が不十分な場合はモデルを上げて再実行する（Haiku → Sonnet → Opus）
- 台本生成（Phase 3）では `/effort xhigh` + Opus 4.8 を推奨（2026-06-05 更新。旧: xhigh + Opus 4.7）。Opus 4.8 は既定が `high` のため xhigh の明示指定が必要

### effort によるトークン最適化（メインセッション）

**現時点の制約**: Agent tool のパラメータに `effort` は含まれない（`model` / `isolation` / `run_in_background` のみ）。サブエージェントへの effort 直接指定は不可。メインセッションの `/effort` 設定はサブエージェントに引き継がれない。

**メインセッションの `/effort` は 5 段階（`low` / `medium` / `high` / `xhigh` / `max`）。v2.1.105+ で xhigh が追加。Opus 4.8 のデフォルトは `high`（Opus 4.7 は `xhigh`）。Opus 4.7 → 4.8 へ切替時に effort が自動的に `high` にリセットされる点に注意。**

**メインセッションの推奨 effort**:

| タスク種別 | 推奨コマンド | 理由 |
|-----------|------------|------|
| 台本生成・設計判断（**Opus 4.8**） | `/effort xhigh`（明示） | Opus 4.8 既定は `high` のため明示指定。コーディング/エージェント/知能重視に最適 |
| 台本生成・設計判断（Opus 4.7） | `/effort xhigh` | Opus 4.7 のデフォルト（legacy 互換） |
| 複雑な実装・レビュー対応（Sonnet 5） | `/effort high` | 品質優先で進める場合の基準値 |
| 通常の実装・PR 対応（Sonnet 5） | `/effort medium` | high よりトークンを節約しつつ実用的な品質を維持 |
| 定型チェック・ルーティング判断 | `/effort low` | 高速・最小コスト |

**サブエージェントのコスト最適化（effort 指定の代替手段）**:

| 手段 | 効果 | 実装方法 |
|------|------|---------|
| **モデル選択（最優先）** | 60〜80% 削減 | `model="haiku"` を積極的に使う |
| **プロンプト指示** | 10〜30% 削減 | 「要点のみ・箇条書き・詳細説明不要」を先頭に記載 |
| **出力凝縮の明示** | 20〜40% 削減 | 「1,000〜2,000 トークンのサマリーで返す」と明示 |

Haiku エージェント向けプロンプト先頭テンプレート:

```
# 出力ルール（必ず守ること）
- 箇条書き・要点のみ（詳細説明・前置き・謝辞は省略）
- ファイル全文でなくパスと行番号のみ返す
- 結果は1,000〜2,000トークン以内
```

## サブエージェントの高度な機能

### ワークツリー隔離（`isolation: "worktree"`）

サブエージェントを一時的な Git ワークツリーで実行し、メインの作業ディレクトリと完全に隔離する。

```
Agent tool 呼び出し例:
  isolation: "worktree"
  prompt: "V008 の画像素材をチェックして..."
```

**活用シナリオ**:
- 画像パイプラインと音声パイプラインを **異なるブランチで並行実行**
- セルフレビューを隔離環境で実行（メインの作業を中断しない）
- 実験的な変更を安全にテスト

**制約**:
- ワークツリー内で変更がなければ自動クリーンアップされる
- 変更があった場合、ワークツリーのパスとブランチ名が結果として返される
- メインのワーキングディレクトリとファイルの競合が発生しない

### バックグラウンドエージェント（`run_in_background: true`）

サブエージェントをバックグラウンドで実行し、完了時に自動通知を受け取る。

**活用シナリオ**:
- ファクトチェックをバックグラウンドで実行しながら台本生成を続行
- AI レビュー依頼後、レビュー到着をバックグラウンドで監視
- theme-discovery のニューススキャンをバックグラウンドで実行

**注意**: バックグラウンドエージェントの進捗を `sleep` でポーリングしない。完了通知を待つ。

### 1M コンテキスト活用ガイドライン

Opus 4.8 / Sonnet 5 の 1M コンテキストウィンドウを効果的に活用するためのガイドライン。

| パターン | 従来（200K） | 1M コンテキスト |
|---------|------------|----------------|
| リサーチ + 台本生成 | リサーチを要約してから台本生成 | リサーチ全文 + キャラ設定 + 既存台本を同時参照可能 |
| レビュー | ファイルごとに分割レビュー | 関連ファイル一括読み込みでコンテキスト横断レビュー |
| パイプライン | 5 ステップごとに手動圧縮 | 10+ ステップでも圧縮なしで続行可能 |

**ただし**:
- 1M あっても不要な情報は渡さない（コスト効率とレイテンシ）
- サブエージェントの出力凝縮ルールは引き続き適用する
- Haiku はコンテキストウィンドウが小さいため、渡す情報を厳選する

## コンテキストコンパクションパターン

長時間実行タスク（Phase 3 台本生成、Phase 4 素材制作など）でコンテキストウィンドウが膨張する問題に対処するため、以下を実践する。

### サブエージェントの出力を凝縮する

- サブエージェントは成果物の全文ではなく **1,000〜2,000 トークンの構造化サマリー** をメインエージェントに返す
- **例外: 台本分割生成**（Phase 3）では、サブエージェントはセクションJSONをそのまま返す（統合のため全文が必要）。ただし1セクション分のみなので肥大化しない
- サマリーのフォーマット例:

```json
{
  "summary": "作業内容の1〜2文での要約",
  "outputs": ["生成したファイルのパス一覧"],
  "key_decisions": ["重要な判断分岐とその理由"],
  "warnings": ["要注意事項（fact_check_flags 件数など）"],
  "next_recommended_action": "次のステップの提案"
}
```

### 長期タスクの検証ポイント設計

エラーは **複合的に蓄積する**（ステップごとの精度 95% × 10 ステップ = 全体成功率約 60%）。
この劣化を防ぐため:

- 5 ステップごと（または Phase 遷移時）に必ず **検証ポイント** を設ける
- 検証は独立したサブエージェント（`model="haiku"`）に委ねて並列化する
- 問題が検出されたら **即座に STOP** してメインエージェントに報告し、続行可否をユーザーに確認する

### パイプライン依存関係（アーキテクチャ制約・プロジェクト例）

> ⚠️ 以下の依存グラフは **出自プロジェクト（動画制作）の実例**。自プロジェクトのパイプライン
> 依存関係に読み替えること（「下流開始前に上流成果物の承認を確認する」原則が本体）。

制作パイプラインには厳密な依存関係がある。下流を開始する前に上流の成果物が「承認済み」であることを確認する。

```
台本JSON（Phase 3）
  └── タイムド台本（Phase 4: voicevox-audio 出力）
        ├── 音声WAV（Phase 4）
        └── 画像素材（Phase 4: image-generator 出力）
              └── Remotionレンダリング（Phase 5）
                    └── YouTube公開（Phase 6）
```

- 上流の成果物が `content/` に存在し、PR がマージ済みであることを確認してから下流を開始する
- Phase をまたぐ作業の開始前に `docs/production-flow.md` の「前提条件」を確認する

## プロジェクト固有オブザーバーエージェントのオブザーバーモード

> ⚠️ **このセクションは出自プロジェクトの実例**。「きなこ」= 技術監修役キャラクター、
> 「もっちー」= 初心者目線チェック役キャラクターの実運用テンプレートをそのまま残している。
> 自プロジェクトでは `.claude/agents/<name>.md` に定義した自分のレビュー役・監修役に
> 名前・禁止事項・チェック観点を置き換えて使う（テンプレート構造＝参加レベル・JSON 出力・
> ゲート昇格条件は汎用）。

### オブザーバーモードとは

> **オブザーバーエージェント**: 結果がワークフローの続行可否（PASS/FAIL）に影響しない。コメント・感想・追加観察を出力するだけ。メインエージェントがその出力を「参考情報」として扱い、無視しても処理が進む。

対義語は「ゲートエージェント」（FAIL 時に次ステップをブロックする）。オブザーバーは基本ノンブロッキングだが、ブランド毀損・技術的誤りを指摘した場合は自動的にゲートへ昇格する。

### 呼び出し側の責任範囲

- `run_in_background: true` で起動してよい（ブロッキング不要）
- オブザーバーが応答しなかった場合でも処理を続行する
- `critical` または `character_issues` に1件以上の指摘がある場合はゲートとして扱い、メインアシスタントが修正する
- オブザーバー結果は Issue コメントまたは PR 説明に「参考情報」として追記する

### @owner（飼い主プロキシ）向け標準呼び出しテンプレート

> **@owner は飼い主の意思決定パターン・美的センス・世界観・センシティブラインを代弁するオブザーバー兼プロダクトオーナー（PO）。** 2 つの面を持つ: ① コンテンツレビュー時は **予測コメントのみ**（A-1〜A-6 の自動承認はしない・本テンプレートの呼び出しはこの面）/ ② PO としては `sp:*` / `priority:*` ラベルの較正を **実行権限付き** で行う（`mcp__github__issue_write`）。権限境界・tools の正本は `docs/rules/session-sprint-rules.md`（§4 PO ロール）と `.claude/agents/owner.md`（SSOT）。`status:*` 操作・Issue クローズ・A-1〜A-6 自動承認はどちらの面でも禁止。

```
参加レベル: Lv{1/2/3}

# 絶対禁止（@owner 設定）
- A-1〜A-6（既約境界外）の自動承認禁止（予測コメントのみ）
- 飼い主の口調・一人称になりきらない（第三者視点で「飼い主はこう反応しそう」）
- 「にゃ」語尾禁止・「AIとして」等の自己開示禁止
- 技術判断（@kinako）・初心者目線（@mochi）の領域に踏み込まない
- 指摘・予測のみ。「修正します」は禁止

# タスク
{確認対象ファイルパスまたはテキスト（台本・SNS投稿・サムネ・メタデータ・企画案）}

# チェック観点
世界観の一貫性・キャラ設定・美的センス・センシティブライン予兆・「飼い主に刺さるか」予測

# 出力フォーマット（Lv1/2/3すべてJSON必須。`sensitive_flags` `worldview_flags` `escalate_to_human` は必ず含める）
```

- `escalate_to_human` に1件以上 → メインアシスタントが飼い主本人への確認フローに回す

### @owner 統合の段階導入ポイント（推奨・順次配線）

> ⚠️ **L-094 desync 防止**: 以下は @owner を差し込む **推奨ポイント** であり、各スキルの SKILL.md への実際の配線は段階導入する。配線が完了したら本表の上段（スキル別統合パターン）に正式行として移し、同一 PR で SKILL.md を更新する。「未配線なのに統合済みと書く」ことを禁止する。

| 優先 | スキル | 差し込みタイミング | @owner の役割（代弁する観点） | ブロッキング性 |
|------|-------|-----------------|--------------------------|--------------|
| P1 | sns-organic-pipeline | Step 4 キャラ監修（@kinako/@mochi）と並列 | 公開前のセンシティブ予兆・世界観・「飼い主に刺さるか」（Lv2） | sensitive/worldview flags でゲート昇格 |
| P1 | sns-publisher | 記事ドラフトレビューチームに追加 | 紹介文の世界観・美的トーン（Lv2） | sensitive/worldview flags でゲート昇格 |
| P2 | script-team-reviewer / screenplay-reviewer | 既存チームと並列 | ナラティブ・キャラ設定・世界観のブレ（Lv2〜3） | worldview flags でゲート昇格 |
| P3 | metadata-reviewer | PASS 動画のスポット監査 | タイトル/サムネの「飼い主に刺さるか」（Lv1） | ノンブロッキング |
| P3 | waiting-user-handler | A 区分提示前 | 「飼い主なら GO/NG しそう」の予測を推奨案に添える（Lv1） | ノンブロッキング（本人承認は不可侵） |

**@kinako / @mochi との並列起動順序**: 台本・SNS では `@mochi（初心者）→ @kinako（技術）→ @owner（世界観・美的・センシティブ）` の3者並列でバランスが取れる。役割（観点）が矛盾した場合は優先順位で打ち消し合わせず、各オブザーバーのゲートシグナル（@kinako の技術 `critical`・@mochi の `character_issues`・@owner の `sensitive_flags` / `worldview_flags`）を **いずれも独立した修正必須シグナル** として扱い、どれか 1 つでも立てば修正する。本人承認が要る項目（`escalate_to_human`）は飼い主への確認フローに回す。

## 注意事項

- 依存関係のあるタスクは逐次実行する（調査結果が実装に必要な場合など）
- サブエージェントの結果はメインエージェントが統合し、ユーザーに簡潔に報告する
- 同じ作業をメインエージェントとサブエージェントで重複して実行しない
