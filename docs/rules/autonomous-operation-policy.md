# CP-6: ユーザー介入最小化と持続可能な自律運用ポリシー

> このドキュメントは CP-6 の詳細実装仕様。**全てのスキル・スケジュールタスク・セッションで適用する。**
> 「ユーザーに判断・作業を最小化し、AI が自律的に開発・制作・マーケティングを持続的に継続できる状態」を設計原則として定義する。

---

## 設計哲学: Human-on-the-loop（監視型）vs Human-in-the-loop（参加型）

### ❌ アンチパターン（Human-in-the-loop）

```
Claude: 実装しました。確認してもらえますか？
User: OK
Claude: PR を作成しますか？
User: はい
Claude: レビューが届きました。対応してよいですか？
User: やって
```

→ ユーザーが **QA チェッカー** になり、24時間対応が必要になる。スループット激減。

### ✅ 正しいパターン（Human-on-the-loop）

```
Claude: [自動実行] 調査 → 実装 → テスト/品質ゲート → PR作成 → AIレビュー対応 → 自動マージ → （必要なら）デプロイ/配信
User: 翌朝、Slack の完了報告を確認するだけ
```

→ ユーザーはシステムを **監視** し、異常時のみ介入する。

## Bounded Autonomy（有界自律性）の定義

### 境界内: Claude が単独で完結させる（ユーザー確認不要）

| カテゴリ | 具体的アクション | 根拠 |
|---------|----------------|------|
| **実装** | 調査・設計・コード変更・リファクタ（要求範囲内） | 定義済みルール |
| **品質ゲート** | セルフレビュー・AI レビュー対応・解消・テスト実行 | `pr-review-flow.md` / `self-reviewer` |
| **PR 管理** | PR 作成・AI レビュー依頼・指摘対応・自動マージ | `pr-review-flow.md` |
| **システム改善** | Retro Try 対応・ルールファイル更新・Lesson 追記 | `retro-try-handler` スキル |
| **リポジトリ衛生** | Stale Issue リセット・Orphan PR 解消 | CP-3 / `project-sync` |
| **障害修正** | CI 失敗・バグの調査と自律修正（L-077 プロトコル） | `problem-investigation-protocol.md` |
| **（ドメイン拡張）** | プロジェクト固有のパイプライン・外部連携・配信等 | プロジェクトで定義 |

### 境界外: ユーザーの最終判断が必要（最小限に絞る）

> **SSOT 参照**: 境界外リストの唯一の正本は `docs/rules/user-confirmation-minimization.md` §1。本表はサマリーであり、確認候補は同ドキュメント §2 の A/B/C/D 分類に必ず落とすこと。「ローカル実行が必要」を理由にした user-work Issue は §4 のクラウド実行試行を経るまで起票禁止。

| アクション | 理由 | 代替自動化 |
|-----------|------|----------|
| `main` への直接 push | 保護ブランチ | PR 経由で自動マージ |
| 外部公開・課金など取り消し困難なアクションの即時手動実行 | 収益・ブランド・コスト影響 | プロジェクト定義の自動スケジュール/ゲートで代替 |
| **致命的な品質ゲート NG** でのパイプライン続行 | 誤り・事故リスク | 品質ゲートで自動 STOP + 通知 |
| サーキットブレーカー発動（2サイクル超） | 無限ループ防止 | 発動後にユーザーに報告 |

> **重要**: 上記の「境界外」リストを **できるだけ短く保つ** ことが CP-6 の目標。判断が増えると運用持続性が低下する。

## 自律実行の優先順位付け

判断に迷ったとき、以下の順序で自律実行を優先する:

```
1. 定義済みルールに該当する → 即実行
2. 類似した過去事例（lessons-core.md / Warm 層）がある → その事例に従い実行
3. リスクが低い（ローカル変更のみ・可逆的）→ 実行して結果報告
4. リスクが中程度（外部システムへの書き込み）→ 実行して Slack 通知
5. リスクが高い（取り消し不可・収益/ブランドに直結）→ Slack でユーザーに確認
```

「5」に該当するケースは **上記の境界外リスト** のみ。それ以外は全て「1〜4」で完結させる。

## タスク投入モード分類（非同期自律 vs 同期監督）

A-1〜A-6（`user-confirmation-minimization.md` §1）は **Claude が実行中に何を確認するか** を定める。本節はそれとは別軸で、**ユーザーがタスクをどの実行モードに投入するか** の積極的な判断ガイド（人間オペレーター向け・Warm 層相当）。Scheduled Tasks や `claude -p` ヘッドレス実行を使うとき、どのタスクを非同期で任せ、どれを同期で見るかを切り分ける。

| 推奨モード | タスク種別の汎用例 | 理由 |
|-----------|------------------|------|
| **非同期自律**（Scheduled Tasks / バックグラウンド・auto-accept 委任） | プロトタイプ実装・周辺/補助機能・ドキュメント整形・定型リファクタ・調査/リサーチ | 失敗しても影響が局所的・チェックポイントから revert できる。"slot machine" 的に任せて結果を受け入れるかやり直す |
| **同期監督**（リアルタイムで設計・品質を確認しながら進める） | コアロジック変更・データ破壊を伴う移行・外部公開系・セキュリティ/認証関連 | 設計判断が多く、誤りのコストが高い。途中で「なぜこの実装？もっと単純にできないか」と介入する余地を残す |

> 「何がコアか」はプロジェクト定義に依存する（`docs/project-mission.md` で具体化）。本表は汎用原則で、機械強制ではなく投入判断の補助に留める。

## ドメイン別 自律化目標（プロジェクトで定義）

本ベースはドメイン非依存。プロジェクト固有のワークフロー（パイプライン）ごとに
「現状の自律度 → 目標自律度 → 残る人手介入ポイント」を `docs/project-mission.md` または
専用ルールに定義し、CP-6 に沿って人手介入を最小化していく。

設計指針:
- 各ワークフロー工程を「境界内（自律実行）／境界外（A-1〜A-6）」に分類する
- 境界外を最小限に保ち、それ以外は実装 → 品質ゲート → PR → AI レビュー → 自動マージまで自律化する
- 外部公開・課金・取り消し困難なアクションのみユーザー確認に回す

### スパム / レート / 予算ガードレール（外部連携時）

外部 API・プラットフォームに自動投稿・自動実行する場合は、プロジェクトごとに
1 日あたりの上限・クールダウン・予算上限を定義し、超過時は自動抑制 or フォールバックする
（`user-confirmation-minimization.md` の D カテゴリ＝外部要因の自律最小化）。

## 持続可能な自律運用の設計パターン

### パターン1: ゼロ-稼働コスト自律化

**原則**: Claude Code の Scheduled Tasks を最大活用し、ユーザーが手を離している間も作業が進む。

```
（例: 時間帯ごとにプロジェクト固有のタスクを割り当てる）
朝   - バックログ消化 / 改善 Issue 実装
日中 - パイプライン実行（プロジェクト定義）
夕方 - リポジトリ衛生（Stale Issue・Orphan PR）+ 日次サマリー
随時 - PR レビュー復帰チェック（全スロット共通プリフライト）
```

> プロジェクトでスケジュール運用する場合は、ルーティング表を専用ルール（例: `hourly-routing.md`）に
> 定義し、各スロット開始時に PR レビュー復帰チェック（`check_pending_pr_reviews.py`）を共通プリフライトとして実行する。

### パターン2: エスカレーション最小化

**原則**: ユーザーへの Slack 通知は「要対応事項 1件以上のときのみ」送信する

```python
# slack_notify.py の実装方針
if actionable_items:
    notify_user(items=actionable_items)  # 要対応があるときのみ通知
else:
    pass  # 全自動完了時は無音でスキップ（ユーザーへの通知なし）
```

### パターン3: セルフ改善ループ

**原則**: 同じミスを繰り返さないよう、自律的にルールを強化する

```
失敗パターン検出
  → retrospective スキルで KPT 生成
  → retro-try-handler が Try を自動実行
  → lessons.md に記録
  → フックまたはバリデーターで Lv3 強制ルール化
```

### パターン4: 状態の永続化（セッション跨ぎ）

**原則**: セッション切れ後も作業を継続できるよう状態を Git に保存する

- `pipeline_state.py` でパイプライン進行状態を管理
- `check_pending_pr_reviews.py` でレビュー待ち PR を次セッションで自動復帰
- Issue コメントに「次回再開ステップ」を記録（session-safety-rules.md 参照）

## ユーザーが最低限やること（週次・月次）

### 週次（週1回・15分以内）

| 作業 | 説明 | 時間 |
|------|------|------|
| Slack ダッシュボード確認 | 週次進捗レポート・アナリティクスをチェック | 5分 |
| PR の diff 確認 | `status:waiting-user` の Issue 対応 | 5〜10分 |
| 成果物の最終品質確認 | 任意 | 任意 |

### 月次（月1回・30分以内）

| 作業 | 説明 | 時間 |
|------|------|------|
| 月次レポート確認 | プロジェクト KPI の進捗を把握 | 10分 |
| マイルストーン確認 | 次フェーズへの移行判断 | 5分 |
| 戦略レビュー | 月次データから方針を調整（任意） | 15分 |

## 禁止パターン（アンチパターン一覧）

以下の行動は CP-6 に違反する。発見したら即 lessons（Warm 層）に記録し、フックで防止する。

| 禁止パターン | 理由 | 対策 |
|------------|------|------|
| 「〇〇してよいですか？」を定義済みルールの範囲で質問する | ユーザーの判断コストを増やす | CP-1 の判断フローチャートを再確認 |
| PRを作成する前に「確認待ち」と報告する | ユーザーがパイプラインの門番になる | 自動パイプラインは Slack 承認をスキップ |
| 検証フラグなしで不確実な数値を成果物に含める | ファクトエラーのリスク | Lv3 フックがブロック |
| セッション切れを理由に作業を放棄する | 持続的運用の妨げ | チェックポイントコミット + 自動復帰 |
| 同じミスを lessons に反映せずに続ける | 同じエラーの再発 | retrospective スキルを必ず実行 |
| **「環境変数がない」「API失敗」等を理由にリサーチ不十分なままユーザー確認に回す（L-077）** | 過去事例の大半が自己解決可能だった。Human-in-the-loop 化 | `docs/rules/problem-investigation-protocol.md` の5ステップ完全実施を必須化 |
| 直接原因だけ修正して中間・根本原因を放置 | 同種エラーが再発 | problem-investigation-protocol.md Step 4（3層因果分析）を必須実施 |
| 1人のエージェントだけで原因特定して停止 | 多角検証なし。視野狭窄 | Agent Teams で3役以上を並列起動（problem-investigation-protocol.md Step 3） |
