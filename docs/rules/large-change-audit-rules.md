# 大規模改善の監査ゲート（Large-Change Audit Rules・SSOT）

> **このファイルは「規模が大きめの改善では、実装 → セルフレビュー → PR で止めず、客観的な検査・監査を必須にする」ルールの唯一の正本（SSOT）である。**
> 飼い主の明示指示（2026-07-19）に基づき新設。背景: サイドパネル UI 改修（PR #113）で、実装後に議論型レビュー・実機検証・新規挙動テストを自動では行わず、ユーザー指摘で初めて実施した。その結果 **セルフレビューでは見逃した実バグ 2 件**（削除時キャッシュ消去欠落・削除後の stale 認証再表示）を検出した。大規模改善では最初からこの監査を通す。
>
> **Warm 層（タスク依存）**: 常駐はしないが、PR フロー（`pr-review-flow.md` / `pr-review-flow-summary.md`）・`pr-review-watcher` / `self-reviewer` スキル・`pre-pr-create-check.sh` フックが本ルールを参照・強制する。`pre-pr-create-check.sh` は PR 作成時に本ルールの監査ディレクティブを機械判定で注入する（見落とし防止）。

---

## 0. 大原則

**「大きめの改善」を実装したら、"既存テストがグリーン" と "自分のセルフレビュー" だけで完了にしない。** 独立した客観レビュー（議論型・敵対的相互検証）と、変えた挙動の実機検証と、その挙動を賭ける専用テストを通してからマージする。過剰検査は小規模変更には課さない（下記の判定基準で対象を絞る）。

---

## 1. 「大規模改善」の判定基準（機械判定: `tools/large_change_audit.py`）

**プロダクト（ランタイム）コードの挙動変更** があり、かつ以下のいずれかに該当するとき「大規模改善」とする:

- 差分（追加 + 削除）が **300 行以上**（`discussion_review_trigger.py` の閾値と一致）、または
- `sp:5` / `sp:8`（大型見積もり）、または
- `type:breaking-change` / `type:security`、または
- プロダクトコードの変更ファイルが **5 個以上**、または
- **新規のユーザー向け機能・UI・認証/データフローの変更**（新しい画面・操作・状態遷移の追加）。

**プロダクトコード** = `src/` ・ `worker(s)/` ・ `functions/` 配下、または `.ts/.tsx/.js/.css/.html` 等のランタイムファイル（テスト `*.spec.* / *.test.* / e2e/`・`docs/`・`.claude/`・`tools/`・`content/` を除く）。

### 免除（監査ゲート対象外・通常フローで可）

- 純粋な `docs/` ・ `docs/rules/` ・ コメント ・ `.claude/`（ハーネス）のみの変更
- テストのみの追加・修正
- 機械的リネーム・自動整形（CJK 整形・lint fix 等）

免除に該当する場合でも、スクリプト・フック・ルールを新設/変更したら **セルフレビュー + 動作確認（self-test・フック発火確認）** は行う（実機で壊れていないことを実結果で確認する・L-113）。免除理由は 1 行記録する。

## 2. 必須監査（大規模改善時・A〜D をすべて満たす）

| # | 監査 | 具体 |
|---|------|------|
| **A** | **議論型レビュー（客観・敵対的相互検証）** | `discussion-review` スキル（ネイティブ Agent Teams）で、対象に応じた複数レンズ（例: UX/デザイン準拠・アクセシビリティ・コード正確性・テストカバレッジ）を独立起動 → 2 ラウンドの相互反論 → lead 統合。`python3 tools/discussion_review_trigger.py` の判定・実行プランに従う。役割分担型 fan-out で代替してよいのはコスト/速度優先を明示した軽微対象のみ（理由 1 行）。 |
| **B** | **挙動の実機検証** | 変えた挙動を実際に動かして観測する（`/verify` 相当）。CI・ユニット・型が緑でも「実際に動くか」は別。UI はスクリーンショット / E2E で、API・データフローは実行結果で確認する。**実結果でのみ断定する（見ていない状態を"通ったはず"で書かない・L-113）。** |
| **C** | **新規挙動の専用テスト追加** | 追加/変更した挙動を賭ける regression gate を書く。「既存テストがグリーン」では不十分（既存は旧挙動しか守らない）。UI なら E2E（開閉・フォーカス・分岐）、ロジックならユニット。議論型レビューで見つかった実バグには、その再発を防ぐテストを必ず添える。 |
| **D** | **議論記録の永続化** | A の議論を `content/discussions/<id>/` に残し（`discussion_whiteboard.py`）、要点（確定した指摘・却下した誤検知・対応）を **PR 説明** と **完了報告** に要約する。 |

> **敵対的検証 + 実機検証の相補性**: 議論型レビューは理論的な指摘（誤検知含む）を出す。B の実機検証がそれを裏取りし、誤検知（例: 「フォーカスが復帰しない」が実機では復帰する）を潰す。両輪で回すこと。

## 3. フロー上の位置づけ（既存 PR フローへの組み込み）

`pr-review-flow-summary.md` の標準フローに、大規模改善のときだけ発火する監査ゲートを重ねる:

```
実装 → セルフレビュー → （大規模判定）→ PR 作成
  → Layer 0 機械ゲート + Layer 1 /code-review（全 PR 必須）
  → 【大規模改善なら】Layer 2 議論型レビュー(A) + 実機検証(B) + 新規挙動テスト(C) + 記録(D) を必須
  → 指摘対応（+ 再発防止テスト）→ 全ゲート通過で自動マージ → 完了報告（監査要点を要約）
```

- `pre-pr-create-check.sh`（PR 作成時フック）が `tools/large_change_audit.py` を呼び、大規模判定時に A〜D のチェックリストと差分から観測した不足（新規テストなし・議論記録なし）を **additionalContext に注入** する（非ブロッキング・見落とし防止）。
- `pr-review-watcher` スキルは大規模改善で Layer 2（A）を **必須実行**（従来は条件付き）。B・C・D の充足も確認してからマージする。
- `self-reviewer` スキルは PR 作成前に大規模判定を行い、C（新規挙動テスト）が差分に無ければ警告する。

---

## 4. 完了・成功の定義

- [ ] `tools/large_change_audit.py --self-test` が PASS
- [ ] 大規模改善では A（議論型レビュー）を実行し `content/discussions/` に記録した
- [ ] B（実機検証）を実結果で確認した（UI はスクショ/E2E・L-113 遵守）
- [ ] C（新規挙動の専用テスト）を追加し、議論で見つかったバグに再発防止テストを添えた
- [ ] D（記録・要約）を PR 説明・完了報告に反映した
- [ ] 免除に該当する変更では理由を 1 行記録した（過剰検査を課していない）
