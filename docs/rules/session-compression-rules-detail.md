## モデル別コンテキストウィンドウサイズ

| モデル | コンテキストウィンドウ | 圧縮発生の目安 | 実用的な上限 |
|--------|---------------------|--------------|------------|
| **Opus 4.8** (`claude-opus-4-8`) | **1M トークン** | ~950K トークン時 | 台本生成・複雑タスク。長時間パイプラインでも圧縮が発生しにくい |
| **Sonnet 5** (`claude-sonnet-5`) | **1M トークン** | ~950K トークン時 | メインセッションで十分な余裕 |
| **Haiku 4.5** (`claude-haiku-4-5`) | **200K トークン** | ~190K トークン時 | サブエージェントに渡す情報を厳選する |

### 1M コンテキストの実運用上の注意

- 1M トークンあっても **不要な情報を詰め込まない**（コスト効率とレイテンシに影響）
- パイプライン実行中に圧縮が発生した場合、PostCompact フックが自動コミットで作業を保護する
- Haiku サブエージェントはコンテキストが小さいため、プロンプトに含める情報を **1,000〜2,000 トークン** に抑える
- 詳細は `docs/rules/claude-code-optimization.md` を参照

## PostCompact 自動コミット機能（内部動作）

### 問題の根本原因

圧縮後に新セッションが起動すると、`session-start.sh` の SessionStart フックが以下を実行する:

```bash
git reset HEAD -- .                                              # ステージ済み変更をリセット
GIT_LFS_SKIP_SMUDGE=1 git checkout -- .                         # 追跡済みファイルの変更を元に戻す（LFS フェッチをスキップ）
git clean -fd -e "assets/" -e "content/pipeline-state/"         # 未追跡ファイルを削除（一部ディレクトリは除外）
```

この「前セッション残留ファイルのクリーンアップ」により、**未コミットのまま残っていた作業内容が全て消える**。

### 解決策: PostCompact での自動コミット

`post-compact.sh` が圧縮直後に以下を自動実行する:

1. クラウド環境（`CLAUDE_CODE_REMOTE=true`）であることを確認（ローカルではスキップ）
2. `main`/`master` 以外のブランチであることを確認
3. 未コミット変更（追跡済み変更・ステージ済み変更・未追跡ファイル）を検出
4. `git add -A` → `git commit -m "[wip] 圧縮前自動コミット（YYYY-MM-DD HH:MM JST）"` → `git push`
5. 成功したことを stdout（セッションコンテキスト）に出力

これにより、次セッションの SessionStart クリーンアップが走っても作業内容がリモートに保存済みとなる。

### 自動コミットの対象

| 状態 | 対象 | 備考 |
|------|------|------|
| ローカル環境（`CLAUDE_CODE_REMOTE` 未設定） | **スキップ** | SessionStart がクリーンアップを実行しないため不要 |
| `main`/`master` ブランチ | **スキップ** | 直接 push 禁止のため |
| その他のブランチ + 変更あり | **自動コミット＆プッシュ** | `[wip] 圧縮前自動コミット` として保存 |
| ワーキングディレクトリがクリーン | **スキップ** | 変更なしのため不要 |

### 自動コミット後の Claude Code の挙動

PostCompact hook の stdout はセッションコンテキストに挿入されるため、Claude が次のログを判断材料として利用できる:
- `✅ 未コミット変更を自動コミット＆プッシュしました` → このメッセージが出ている場合は、そのコミットを前提に作業を継続し、追加のコミットは不要と判断する
- `✅ 自動コミット完了（プッシュ失敗）` → このメッセージが出ている場合は、`git push` をコマンドで実行する対応が必要と判断する

### wip コミットの扱い

自動コミットされた `[wip]` コミットは、作業再開後に通常のコミットに squash するか、
そのまま PR に含めて AIレビュー後マージするかをケースバイケースで判断する。

## 圧縮プロンプトの品質設計（Context Budget Design・T27）

圧縮（コンパクション）は **コンテキストウィンドウの節約** であると同時に、**情報の取捨選択** でもある。圧縮後の動作品質は「何を圧縮サマリーに残すか」に大きく依存する。

**圧縮に強いコンテキスト設計の原則**:

| 原則 | 理由 | 実装方法 |
|------|------|---------|
| **重要な決定と結果を明示コミットする** | 圧縮サマリーは「なぜそうなったか」より「何が決まったか」を優先保持する | Issue/PR コメント・wip コミットメッセージに決定事項を書く |
| **中間思考ではなく結論を Issue に記録する** | 長い思考ログは圧縮でカットされる。Issue コメントは圧縮後も `mcp__github__issue_read` で取得可能 | 「次回再開ステップ」を Issue コメントに記録（`session-safety-rules.md`） |
| **長大な入力データをコンテキストに置かない** | 圧縮サマリーに入力全文が残ると次セッションのコンテキストを浪費する | 大きなファイルは Read 都度アクセス・コンテキストに置き続けない |
| **サブエージェントへの委譲結果を要約で返す** | サブエージェント全文を親コンテキストに置くと膨張する | サブエージェントは 1,000〜2,000 トークンのサマリーを返す（`agent-team-summary.md`） |

**L-101（`tool call could not be parsed`）との関係**: L-101 は大コンテキスト + 長い thinking の組み合わせで発生しやすい（`lessons-core.md`）。圧縮品質が悪いと圧縮後のセッションが直後に再圧縮を繰り返す（サマリーが肥大していると圧縮効果が薄い）。上記の原則でコンテキスト品質を保つことが根本対策。

## シンボリックリンク自動修正の詳細

同期漏れは以下の 2 つのタイミングで自動検出・修正される:

| タイミング | 検出方法 | 修正方法 |
|-----------|---------|---------|
| **セッション開始時** | `session-start.sh` が `check_rules_sync.sh --fix` を実行 | 漏れたシンボリックリンクを自動作成 |
| **圧縮後** | `post-compact.sh` が `check_rules_sync.sh --fix` を実行 | 漏れたシンボリックリンクを自動作成 |

自動修正は git 管理外の変更（ローカルリポジトリの symlink 作成）を行うため、セッションごとに実行しても安全。

### 手動確認コマンド

```bash
# 同期状態の確認（終了コード 0 = OK、1 = 不足あり）
./tools/check_rules_sync.sh

# 不足を自動修正
./tools/check_rules_sync.sh --fix
```
