# ユーザー指示のIssue化ルール

ユーザーから作業指示を受けたとき、 **Claude が自律的にIssueを作成して記録する** ためのルール。
過去の指示・対応履歴をIssueで追跡可能にし、コミット履歴やPRを遡るコストを削減する（CP-6）。

> **原則**: Issue は「なぜ変更したか」を記録する唯一の場所。コミットメッセージは「何をしたか」を記述するが、「ユーザーがどういう意図で依頼したか」はIssueにしか残らない。

---

## Issue化の判断フロー

```
ユーザー指示を受信
  ↓
ファイル変更・コード変更・設定変更を伴うか？
  ├── No → 純粋な質問・説明・確認のみ → Issue化不要（即回答）
  └── Yes ↓
        作業完了に 5 分以上かかるか / 複数ファイルに影響するか？
          ├── No（明らかな1行修正・誤字修正等）→ Issue化不要（即実行して完了報告）
          └── Yes → Issue化してから実行（下記フロー）
```

### Issue化してから実行するフロー

```
1. Issue を作成（タイトル・本文・ラベルを付与）
2. Issue に `status:in-progress` を付与（CP-4 論理ロック）
3. 作業ブランチを作成して実装
4. PR 作成 → AIレビュー → 自動マージ
5. Issue をクローズ（Closes #N を PR 本文に記載）
```

## Issue化が必要なケース（記録すべき指示）

| カテゴリ | 具体例 |
|---------|--------|
| **コード・スクリプト変更** | ツール修正・新スクリプト追加・既存コード改修 |
| **ルール・設定変更** | CLAUDE.md 更新・`docs/rules/*.md` 変更・`settings.json` 変更 |
| **スキル追加・変更** | SKILL.md の新規追加・既存スキルの改修 |
| **パイプライン改善** | script/audio/image/video パイプラインへの機能追加・修正 |
| **新機能・改善依頼** | 「〜できるようにして」「〜を改善して」 |
| **バグ修正依頼** | 「〜が動かない」「〜がおかしい」 |
| **ドキュメント更新** | `docs/*.md` 新規作成・大幅更新（誤字修正は除く） |
| **複数ファイルにまたがる変更** | リファクタリング・命名変更・構造変更 |

## Issue化不要なケース（記録不要な指示）

| カテゴリ | 具体例 |
|---------|--------|
| **純粋な質問・調査** | 「〜について教えて」「〜を調べて」（ファイル変更なし） |
| **動作確認・レビュー** | 「〜を確認して」「〜をチェックして」（ファイル変更なし） |
| **自明な1行修正** | 誤字修正・コメントの軽微な修正（1ファイル・1箇所・30秒以内） |
| **セッション内の一時的な作業** | 動作確認のための一時的なコマンド実行 |

> **迷ったらIssue化する** 。Issue作成コストは低く、記録がないことのコストは高い。

## Issueのタイトル・ラベル規約

### タイトルプレフィックス

| 指示の種類 | タイトル形式 |
|-----------|------------|
| 新機能・機能追加 | `feat: {概要}` |
| バグ修正 | `fix: {概要}` |
| 改善・リファクタリング | `improvement: {概要}` |
| ドキュメント・ルール更新 | `docs: {概要}` |
| 動画制作関連 | `[V{ID}] {Phase}: {概要}` |

### 必須ラベル

| ラベル | 条件 |
|--------|------|
| `type:feature` | 新機能・新スキル追加 |
| `type:bug` | バグ修正 |
| `type:improvement` | 改善・リファクタリング |
| `type:docs` | ドキュメント・ルール更新 |
| `status:in-progress` | 作業開始時に即付与（CP-4 論理ロック） |
| `sp:N` | 見積もり SP を必須付与（1/2/3/5/8。基準は `docs/rules/session-sprint-rules.md` §3。done_sp 計測のため省略禁止） |

## Issue本文テンプレート

```markdown
## ユーザー指示

{ユーザーから受けた指示の内容（原文に近い形で記録）}

## 対応方針

- {何をするか（箇条書き）}

## 完了条件

- {何ができたら完了とするか}
```

作成は `mcp__github__issue_write`（create アクション。クラウド一次経路・L-114）または `gh issue create --label "type:docs,status:in-progress"` を使う。実例は `user-instruction-issue-rules-detail.md` を参照。

---

## 禁止パターン

- ユーザーが「Issueにして」と言わないと Issue を作らない → **禁止** （このルール違反）
- Issue を作らずにコミットだけする → **禁止** （追跡不能）
- Issue タイトルに「ユーザーに言われたから」と書く → **禁止** （自明なため不要）
- 質問・確認のたびに Issue を作る → **禁止** （記録不要なケース）

---

## スケジュールタスク・自律実行との関係

スケジュールタスク（`{プロジェクト定義: hourly-routing 相当}`）や自律実行パイプラインが起動した場合は、
各パイプラインスキル（SKILL.md）内の Issue 管理フローに従うため、本ルールは **適用しない** 。
本ルールは **ユーザーが直接発した指示** に対してのみ適用する。
