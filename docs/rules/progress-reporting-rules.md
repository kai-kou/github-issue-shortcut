# 制作系長時間処理の進捗報告ルール（SSOT）

> **このファイルは「画像・音声・動画など制作系の長時間処理における進捗報告」の唯一の正本（Single Source of Truth）である。**
> **Warm 層（タスク依存・E-B #20）**: 制作系の長時間処理（音声・画像・動画レンダリング等）を行うパイプラインスキルが **冒頭で Read** する。全セッション常駐（Hot）ではない。
>
> **目的（2 つを同時に満たす）**:
> 1. **セッションタイムアウト防止** — 長時間処理中に出力が途切れると「Stream idle timeout」「Request timed out」やアイドル切断が発生する。適切な頻度で stdout を出すことで防ぐ。
> 2. **進捗確認** — ユーザー（および Claude 自身）が「今どこまで進んだか・残りどれくらいか」を把握できる。

---

## 0. 適用範囲（いつこのルールを発動するか）

制作系パイプラインで **想定実行時間が 5 分を超える処理** を実行するときに必ず適用する。

下表はプロジェクト定義の制作ツールでの **代表例** （プロジェクトに応じて読み替える）。

| 処理（例） | 代表ツール / スキル（プロジェクト定義） | 想定時間 | ボトルネック |
|------|------------------|---------|------------|
| 音声生成 | プロジェクト定義の音声生成ツール | 15〜40分 | アイテム数ぶんのループ |
| 画像生成 | プロジェクト定義の画像生成ツール | 20〜60分 | アイテム数ぶんのループ |
| BGM / 音響生成 | プロジェクト定義の音響生成ツール | 5〜15分 | シーン数ぶんのループ |
| サムネイル / 静止画生成 | プロジェクト定義の画像生成ツール | 5〜10分 | 試行回数 |
| 動画レンダリング | プロジェクト定義のレンダリングツール | 30〜90分 | 単一の長時間コマンド |

> 5 分以下の処理（PR 作成・メタデータ生成・ラベル更新等）はこのルールの対象外（通常のステップ報告で十分）。

---

## 1. 進捗報告の 3 つの軸（どう報告するか）

処理の形によって使う軸が異なる。**該当する軸を全て使う**。

### 軸 A: アイテム単位報告（ループ処理 — 画像・音声・音響生成等）

Claude が **MCP ツールやスクリプトをループで複数回呼ぶ** 処理（例: 画像を 1 枚ずつ生成ツールで生成）では、**5 アイテムごと、または進捗 25% ごとのどちらか早い方** で 1〜2 文の進捗テキストを出力する。

```
🎨 [{ID} image] 画像生成中 12/48 枚（約25%・経過 8 分）にゃ。次の塊いくにゃ
```

- 各アイテム呼び出しの **戻り値を受け取った直後**、次のツール呼び出しより前にテキストを出す（サイレント思考の連続を避ける・L-055 と同根）
- 1 アイテムが重い場合（1 枚 30 秒超）は **毎アイテム** 報告してよい

### 軸 B: 時間単位報告（heartbeat — 単一の長時間コマンド）

音声生成・レンダリングのように **1 回の Bash コマンドが 5 分を超える** 処理は、フォアグラウンドで待つと Bash ツールのタイムアウト（デフォルト 120 秒・最大 600 秒＝10 分）を超える。必ず以下の構成にする。

```
1. 本体を run_in_background で起動し、ログを /tmp/{id}.log にリダイレクト
2. tools/progress_heartbeat.sh を run_in_background で起動（3 分間隔で stdout 出力）
3. 返った heartbeat の PID を Monitor ツールに渡して stdout をストリーム
   → 3 分ごとのハートビート行が来るたびにセッションが維持され、進捗も見える
4. 完了/エラーパターンを heartbeat が検出したら自動終了 → 本体の結果を確認
```

> `tools/progress_heartbeat.sh`（汎用）と `tools/render_heartbeat.sh`（レンダリング専用・完了パターン内蔵）のどちらでもよい。レンダリングは render_heartbeat、それ以外（音声生成等）は progress_heartbeat を使う。

### 軸 C: ステップ/フェーズ単位報告（チェックポイント）

各ステップ・フェーズの完了時に、**wip コミット + 1〜2 文の進捗テキスト** を出す（既存の中間コミット運用・`checkpoint` スキルと連携）。

```
✅ [{ID} audio] Step 4 音声生成完了（48 アイテム・実測 16 分 12 秒）。次は Step 5 チームレビューにゃ
```

- これにより、セッションが切れても次セッションが wip コミットと Issue コメントから再開できる（session-safety-rules.md）

## 2. 進捗報告フォーマット（統一）

stdout / チャット向けの 1 行フォーマット:

```
{emoji} [{ID} {pipeline}] {処理名} {進捗}/{全体}（約{％}・経過{N}分）{コメント} ({HH:MM JST})
```

| pipeline | emoji |
|----------|-------|
| audio | 🎙️ |
| image | 🎨 |
| video / render | 🎬 |
| bgm | 🎵 |

- 進捗が数えられない処理（単一コマンド）は `{進捗}/{全体}` を省略し経過時間と最終ログ行を出す
- 完了は `✅`、エラーは `❌`、タイムアウトは `⏰` を先頭に付ける

## 3. Slack 進捗確認（ユーザー向け・過多防止）

ユーザーが手元を離れていても進捗を追えるよう、**長時間処理（想定 15 分超）の開始時と完了時に Slack 通知** する。中間通知は原則しない（CP-6 のエスカレーション最小化と整合）。

進捗通知は FYI 扱い（`@mention` しない）なので、専用の重い `progress` タイプ（制作進捗レポート・@mention 付き）ではなく **`message` タイプ** を使う。

```bash
# 開始時（任意・想定15分超の処理のみ）— @mention しない FYI
python3 tools/slack_notify.py message --text "🚀 [進捗 audio] {ID} 音声生成開始（48 アイテム）"
# 完了時
python3 tools/slack_notify.py message --text "✅ [進捗 audio] {ID} 音声生成完了（実測16分）"
```

> **推奨**: `tools/progress_heartbeat.sh --slack --pipeline audio` を使えば、開始・完了・エラーのマイルストーン Slack 通知を heartbeat 側が上記 `message` タイプで自動代行する。`slack_notify.py` はトークン未設定なら無音スキップ（既存仕様）なので、ローカル/未設定環境でも安全。
>
> **異常時のみ @mention**: 進捗通知（progress）は完了報告系であり `@mention` しない（`user-notification-triage.md` の FYI 扱い）。真にユーザー操作が必要な障害（A-1〜A-6）に発展した場合のみ triage を経て `@mention` する。

## 4. 標準フロー（音声・レンダリングの雛形）

```
# ── 単一長時間コマンド（音声生成 / レンダリング）の標準形 ──
1. （任意）Slack: progress start 通知
2. Bash(run_in_background=true): {本体コマンド} > /tmp/{id}.log 2>&1
3. Bash(run_in_background=true): bash tools/progress_heartbeat.sh {id} {max分} --done "完了パターン" --error "エラーパターン" [--slack]
4. Monitor: heartbeat の PID を監視（3分ごとのハートビート行でセッション維持＆進捗確認）
5. 完了検出 → 本体ログで成否確認 → wip コミット + ✅ 進捗テキスト
6. （任意）Slack: progress done 通知
```

```
# ── ループ処理（画像生成）の標準形 ──
1. 全体件数 N を把握し「これから N 枚生成するにゃ」と宣言
2. 5 枚ごと（または各 25%）に軸 A の進捗テキストを出力
3. 区切りごとに wip コミット（パイプライン既存運用）
4. 完了 → ✅ 進捗テキスト
```

## 5. 禁止 / 推奨パターン

```
❌ 40 分かかる音声生成を foreground の単一 Bash で実行（Bash タイムアウト超過・出力途切れで切断）
❌ 48 枚の画像をサイレントに連続生成し、完了まで一切報告しない
❌ レンダリング中に出力ゼロで 10 分以上待つ（アイドルタイムアウト）
❌ 進捗のたびに毎回 Slack @mention（通知過多・CP-6 違反）
✅ run_in_background + progress_heartbeat + Monitor で 3 分ごとに進捗を出す
✅ ループは 5 件 / 25% ごとに進捗テキスト + 区切りで wip コミット
✅ Slack は開始・完了の 2 点のみ（@mention なし FYI）
```

## 7. 完了・成功の定義

- [ ] 制作系パイプライン（音声/画像/動画/レンダリング等・プロジェクト定義）が本ルールを参照し、長時間処理で進捗報告する
- [ ] 単一長時間コマンドが run_in_background + heartbeat + Monitor で監視され、アイドルタイムアウトを防止できる
- [ ] ループ処理が 5 件 / 25% ごとに進捗テキストを出す
- [ ] 進捗が（想定 15 分超は Slack でも）確認できる
- [ ] `progress_heartbeat.sh --self-test` が PASS
