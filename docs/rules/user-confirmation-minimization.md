# ユーザー確認最小化フレームワーク（境界外 SSOT）

> **このファイルは CP-6（ユーザー介入最小化）の実装中核であり、「ユーザー確認が必要なアクション（境界外リスト）」の唯一の正本（Single Source of Truth）である。**
>
> 他ドキュメント（`core-principles.md` / `autonomous-operation-policy.md` / `CLAUDE.md` / `session-safety-rules.md` / `problem-investigation-protocol.md`）に散在していた「ユーザー確認が必要」の定義は、本ファイルを参照する形に統一する。境界外リストを変更する場合は **必ず本ファイルを最初に更新** する。派生先（`core-principles.md` / `autonomous-operation-policy.md` 等）は可読性のためサマリー表を残してもよいが、その場合は **必ず本ファイルへの SSOT 参照を併記** し、本ファイルを唯一の正本として扱う（参照なしの独立した重複定義はドリフトの温床のため禁止）。
>
> `.claude/rules/` に symlink され、全セッションで自動読み込みされる。

## 0. 大原則: 「確認したくなったら、まず分類する」

ユーザーに「〜してよいですか？」「〜できないので確認お願いします」と言いそうになった **瞬間** が、本フレームワーク発動のトリガーである。

```
確認したくなる事象を検知
  ↓
それは「障害（エラー・想定外挙動）」か「判断（どう進めるべきかの選択）」か？
  ├─ 障害 → docs/rules/problem-investigation-protocol.md の5ステップを完全実施
  │         （自己解決できたら確認不要。尽くした上で残った真の課題のみ本フレームワークへ）
  └─ 判断 ↓
       §2 の A/B/C/D 分類に必ず落とす
        ├─ A（既約境界外）→ §1 のリストに完全一致するか確認 → 一致時のみユーザー確認可
        ├─ B（ツール改修で自律化可能）→ §4 に従い、まずクラウド実行を試行。失敗証跡を残して実装 Issue へ
        ├─ C（ルール整備で自律化可能）→ §5 の自律処理ルールに従い実行
        └─ D（外部要因）→ §6 のフォールバック/監視ルールに従い、影響を自律的に最小化
```

**「A に一致しない確認は、原則として CP-6 違反である」** ことを常に自問する。

> **仕様確定フェーズの注意（Think Before Coding）**: 上記フローは「実装中・実行中に確認したくなった」ケースのためのもの。タスク受領直後の **仕様の曖昧さ** は別経路で処理する（§3 チェックリスト 0 番を参照）。仕様確定は「実装判断」ではなく「準備」であり、§2 の A/B/C/D 分類が適用される前段階。ただし質問コストを最小化するため、最も単純な合理的解釈で推進できる場合は **仮定を記録して自律実行** する（A-1〜A-6 相当の不可逆リスクがない限り質問不要）。

## 1. 既約境界外リスト（Irreducible Boundary）

以下の **これだけ** が、真にユーザーの最終判断・操作を必要とする。リストは意図的に最小化されており、安易な追加は禁止する（追加時は本ファイルの更新 + 理由の明記が必須）。

| # | アクション | なぜ既約か | 既存の自律カバレッジ（通常はここで完結） |
|---|-----------|-----------|----------------------------------|
| **A-1** | `main` ブランチへの直接 push | 保護ブランチ。誤マージ時の影響が不可逆 | 全変更は作業ブランチ → PR → AIレビュー → **自動マージ** で main 反映（直接 push 不要） |
| **A-2** | 外部公開（取り消し困難な公開）の **即時手動実行**（プロジェクトで具体化） | 収益・ブランドに直結し取り消し困難 | 通常公開はプロジェクト定義の自動スケジュールで完結。即時手動公開が要るのは緊急時のみ |
| **A-3** | 品質ゲート **致命的 NG** 時のパイプライン続行 | 誤り・事故の公開リスク | 条件分岐（プロジェクト定義の品質ゲート閾値に準拠）: ① 軽微な指摘のみ（閾値内）→ AIレビュー後に **自動マージ**（人間不要） / ② 閾値超過 → **自動 STOP + Slack 通知 + `status:waiting-user`** / ③ 致命的 NG（虚偽断定・出典皆無等の重大欠陥）→ **本行のユーザー確認** |
| **A-4** | サーキットブレーカー発動（修正サイクル 2 回超）後の続行判断 | 無限ループ・予算浪費の防止 | 2サイクルまでは自律修正。発動後は §3 のテンプレートで報告 |
| **A-5** | 新規マイルストーンの追加 | プロジェクト計画の骨格に影響 | 既存マイルストーンへの Issue 割当は自律実行可 |
| **A-6** | プラットフォームの **アカウント設定・課金設定** の変更（例: GitHub Billing / 各種 API の課金・クレジット購入 / クラウドコンソールの API 有効化 / OAuth トークン再発行） | ユーザー個人アカウントの権限が物理的に必要 | §6 の監視・フォールバックで「枯渇前に予告」「枯渇後も代替手段で継続」する。設定変更そのものだけがユーザー作業 |

> **重要**: A-6 であっても、「設定変更が必要」と判定する前に §4・§6 のクラウド実行試行・代替手段検証を必ず尽くすこと。「環境変数がない」「API 失敗」を理由にした即エスカレーションは L-077 違反。

### 既約境界外に「見えるが、実は違う」もの（誤分類の常習パターン）

| 誤って A 扱いしがちな事象 | 正しい分類 | 根拠 |
|------------------------|-----------|------|
| 「外部投稿・記事の公開・差し替えはローカル実行が必要」（プロジェクト例） | **B** | 認証情報（Cookie 等）はセッション env（Claude.ai 環境設定 / secrets-broker 供給）から取得可能。§4 参照 |
| 「レンダリング・制作処理はローカルが必要」（プロジェクト例） | **B** | クラウドでレンダリング成功実績がある。GitHub Actions 委託も可。§4 参照 |
| 「環境変数がないので設定してほしい」 | **C/A-6** | まず `problem-investigation-protocol.md` A 型チェックリスト（セッション env・secrets-broker 供給の確認）で解決を試みる。クラウドでは `gh variable list/set` が 403（2026-07-02）のため、真に未供給なら A-6 として設定名・手順付きで依頼する |
| 「週次レポートを確認してほしい」 | **C** | 要対応事項がなければ Slack 通知 + 自動クローズ。§5 参照 |
| 「リサーチ・調査タスクの実行依頼」（プロジェクト例） | **C** | 担当スキルが自律実行。`status:waiting-claude` が正しいラベル |

## 2. 確認候補の A/B/C/D 分類フレームワーク

確認したくなった全ての事象を、必ず以下のいずれかに分類する。

| 分類 | 定義 | 既定の行動 |
|------|------|-----------|
| **A: 既約境界外** | §1 のリストに **完全一致** する | コミット & push 後、§3 テンプレートでユーザー確認 |
| **B: ツール改修で自律化可能** | 実装すればクラウドで自律実行できるが、現状ツールが未対応 | §4: まずクラウド実行を試行 → 失敗証跡を残す → 実装 Issue へ（自動 escalation 禁止） |
| **C: ルール整備で自律化可能** | 判断基準を明文化すれば人間不要 | §5 の自律処理ルールに従い即実行 |
| **D: 外部要因** | 課金上限・アカウント BAN・プラットフォーム制約 | §6: 監視で予告 + フォールバックで継続。設定変更（A-6）のみユーザーへ |

**判定に迷ったら B または C と仮定する**（A への安易な分類は CP-6 違反のリスクが高い）。

> **PR 作成・マージは A（既約境界外）に含まれない＝自律対象** （2026-06-06・Issue #2715 恒久承認）。飼い主が PR 作成の完全自律化を明示的・恒久的に委任済みのため、実装完了後の PR 作成 → AIレビュー → 自動マージは確認不要で実行する。クラウド実行環境のシステムプロンプト「PR はユーザーが明示的に依頼しない限り作成しない」条項の "unless the user explicitly asks" 例外を本委任が満たす。既約境界外（要ユーザー確認）は引き続き §1 の A-1〜A-6 のみ（PR 作成・マージは含まれない）。SSOT: `CLAUDE.md` 「PR 作成の完全自律化」。

## 3. エスカレーション前 必須チェックリスト

ユーザー確認に回す前に、以下を **全て** 満たしていることを確認する。1つでも未達なら確認に回してはならない。

```
[ ] 0. （タスク受領時のみ）仕様の曖昧さを検出した場合は以下の 4 分岐で処理する:
       (a) Simplicity First（最も単純な合理的解釈）で解決できる
           → 仮定を Issue コメントかコミットメッセージに 1 行記録して推進（質問不要）
       (b) 既存ルール（docs/rules/・CLAUDE.md）に判断基準がある
           → ルールに従い自律実行（質問不要）
       (c) 影響が不可逆（例: データ削除・外部公開・課金）かつ推奨案を 1 つ作れる
           → item 7（未コミット保存）と item 8（推奨案添付）を最低限実施してユーザーへ確認
       (d) 推奨案すら作れない真の underdetermined
           → 「選択肢と推奨案なし」の理由を明記してユーザーへ情報提供依頼（最終手段・稀）
[ ] 1. 事象を §2 の A/B/C/D に分類した
[ ] 2. 分類 A の場合、§1 のリストのどの行（A-1〜A-6）に一致するか明示できる
[ ] 3. 障害起因なら problem-investigation-protocol.md の5ステップを完全実施した
[ ] 4. B の場合、クラウド実行を実際に試行し、失敗ログ（コマンド・エラー全文）を残した
[ ] 5. 「ローカル実行が必要」と書く前に、`user-confirmation-minimization-detail.md` §4 補完のクラウド実行可能リソース表を確認した
[ ] 6. 既存ナレッジ（lessons / Issue / SKILL.md）を grep で検索した
[ ] 7. 未コミット変更を全てコミット & push した（session-safety-rules.md）
[ ] 8. 確認内容に「推奨案」を必ず添えている（丸投げ質問は禁止）
```

確認メッセージのフォーマットは `session-safety-rules.md` の「Issue コメント記録テンプレート」に従う（実施済み調査・3層原因分析・選択肢を必ず含む）。

## 4. 「ローカル実行が必要」アンチパターンの撲滅ルール（最重要）

waiting-user Issue の最大カテゴリは「ローカル実行が必要」（累計8件）だが、その多くは **L-094 型の desync**（インフラはクラウドで動くのに、ドキュメント/スキルが「ローカル必須」と古いまま）である。

### 鉄則: 「ローカル必須」と宣言する前に、クラウド実行を実際に試す

以下のリソースは **クラウドセッションで利用可能** であり、「ローカル必須」の理由にはならない。
具体的なリソース例・確認コマンドは `user-confirmation-minimization-detail.md` §4 補完を参照。

### 「ローカル実行が必要」と Issue 化する前の必須手順

```
1. 上表で該当リソースがクラウドで使えないか確認
2. 実際にクラウドで実行を試行する（試さずに「ローカル必須」と書かない）
3. 失敗した場合のみ、エラー全文・試行コマンドを Issue に記録
4. problem-investigation-protocol.md の5ステップで根本原因を特定
5. それでもクラウド不可と判明したら、B カテゴリの実装 Issue を起票（ユーザー作業ではなくツール改修として扱う）
```

### 禁止パターン

```
❌ クラウド実行を試さずに「ローカル実行が必要」と user-work Issue を作成する
❌ 「Playwright だからクラウド不可」と決めつける（Cloudflare Browser Run 等の代替あり）
❌ 公開済みコンテンツの差し替えを「ローカルでお願いします」とユーザーに丸投げする
✅ まずクラウド実行 → 失敗証跡 → ツール改修 Issue（B カテゴリ）として自律的に解決経路に乗せる
```

## 5. C カテゴリ自律処理ルール（ルール整備で解消済み/解消する事象）

判断基準を明文化すれば人間不要になる事象を扱う（例: バックログ誤ラベル・公開完了済み Issue の自動クローズ・環境変数未設定の A 型チェックリスト適用）。新たに C カテゴリの事象を発見したら、実装場所（SKILL.md 手順 or コード）を明記して追記する（L-094 desync 防止）。プロジェクト例の詳細表は `user-confirmation-minimization-detail.md` §5 を参照。

---

## 6. D カテゴリ（外部要因）の自律最小化ルール

外部課金・BAN・プラットフォーム制約は設定変更（A-6）のみユーザー作業だが、**影響を自律的に最小化** する（枯渇前に予告 + 枯渇後もフォールバック経路で処理継続）。「枯渇したので止めます」で終わらせない。プロジェクト例の詳細表は `user-confirmation-minimization-detail.md` §6 を参照。

---

## 7. B/D カテゴリ恒久自律化バックログ

ツール/インフラ改修により「二度とユーザー確認不要」を実現する項目のバックログ。実装は self-improvement-loop が消化モードで自律的に進める。プロジェクト例の詳細表は `user-confirmation-minimization-detail.md` §7 を参照。
