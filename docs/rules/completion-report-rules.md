# セッション完了報告ルール（Completion Report Rules・SSOT）

> **このファイルは「セッション対応が完了したときの、ユーザー向け完了報告の構造」の唯一の正本（SSOT）である。**
> `.claude/rules/` に symlink され、全セッションで自動読み込みされる（Hot 層・圧縮後も有効）。
> 既存の L-052 / L-076 / L-102（`pr-review-flow.md` ・ `pr-review-flow-summary.md`）はアウトカムの「中身」を規定し、
> 本ファイルは完了報告の「構造（並び順・何を主役にするか）」を規定する。両者は一対で機能する。

---

## 0. なぜ必要か（ユーザー実体験・2026-06-14）

飼い主からの明示の指摘:

> 「セッション対応完了した際に、 **PR マージの詳細ではなく、セッション当初の指示に対する完了報告（特にアウトカム中心）** を出してほしい。
> 『PR マージこんな感じでマージしました！』だと『で、何をお願いしてたっけ？』と **チャットの最初までさかのぼる** ケースが多くて困る。」

**根本原因**: 完了報告が「PR マージ手順・レビュー往復・差分の説明」を主役にしてしまい、 **ユーザーが最初に何を頼んだか（初回指示）と、その結果どうなったか（アウトカム）が報告に欠落** していた。ユーザーがチャットを遡らないと文脈を取り戻せない報告は **報告設計の欠陥** である。

---

## 1. 完了報告の必須構造（この並び順で出す）

セッション対応が完了したら、ユーザー向けチャットの **最終メッセージ** を必ず以下の構造で出す。

```markdown
## ✅ 完了報告

**ご依頼**: {セッション開始時の指示を1文で再掲（ユーザーが遡らなくて済むように）}
**できるようになったこと**: {アウトカムを1〜2文。「何ができるようになったか（結果）」を書く}

（必要な場合のみ）**主な変更**: {1〜3行。要点だけ。手順ではなく要点}

---
[PR #{N}]({URL}) / ブランチ `{branch}`  ← 末尾に1行の補足。見出しにしない
```

> **URL 記法の注意（L-112）**: bare URL `（URL）` 形式は禁止。`[PR #N](URL)` の Markdown リンク記法を必ず使う。
> bare URL の直後に `・` 等の Unicode 文字を置くと URL パーサーが誤って URL に取り込み 404 になる（再発防止）。

### 鉄則

1. **冒頭は必ず「ご依頼」の再掲**。これがあるだけでユーザーはチャット先頭まで遡らずに済む（本ルールの最重要目的）。
2. **主役はアウトカム（結果）**。「何をしたか（プロセス）」ではなく「何ができるようになったか」を書く（L-052）。
3. **初回指示に紐づける**（L-076）。PR 対応が長引いても、最終報告は「最初の指示に対して何が達成されたか」で締める。
4. **PR 番号・URL・ブランチ・マージ方法は末尾の補足1行**。完了報告の主役にしない。
5. **プロセスの詳細は書かない**（L-102 サイレント原則）: AI レビュー往復・指摘件数・修正サイクル・squash/rebase 等のマージ手順・コミット数は完了報告に含めない（記録は PR スレッド・Issue コメントに残る）。

## 2. 良い例 / 悪い例

### ✅ 良い例

```markdown
## ✅ 完了報告

**ご依頼**: 完了報告を「PR マージ詳細」ではなく「初回指示のアウトカム中心」にするようルール・スキル・ハーネスを調整。
**できるようになったこと**: セッション完了時に、最初の依頼の再掲とアウトカムが冒頭に出るようになり、チャットを遡らずに「何を頼んで何が達成されたか」が一目で分かるようになったにゃ。

---
[PR #36](https://github.com/.../pull/36) / ブランチ `claude/...`
```

### ❌ 悪い例

```markdown
PR #36 を squash でマージしました！Gemini の指摘 3 件と Copilot レビューを解消、修正サイクルは 2 回でした。
completion-report-rules.md と CLAUDE.md と check_rules_sync.sh を更新しています。
```

→ 初回指示の再掲なし・アウトカムなし・プロセス（マージ方法/レビュー往復/ファイル名）が主役。 **これが「で、何頼んだっけ？」を生む** 。

## 3. Slack 完了通知との関係

`tools/slack_notify.py pr --pr-title "[完了] ..." --outcome "{1文}"` の `--outcome` も本ルールに従う。

- `--outcome` は「初回指示に対して何ができるようになったか」1文（L-076）。
- プロセス的文言（「マージしました」「PR #N」「〜を更新した」「指摘 N 件対応」等）は入れない。
- `slack_notify.py` は `[完了]` 通知で `--outcome` が空 or プロセス的文言の場合に stderr 警告を出す（ハーネスガードレール）。

## 4. 例外（プロセスに触れてよい / 触れるべきケース）

| 事象 | 対応 |
|------|------|
| サーキットブレーカー発動（修正サイクル 2 回超・A-4） | 完了ではなく中断。`problem-investigation-protocol.md` のエスカレーションテンプレートで報告 |
| ファクトチェック致命的 NG（A-3） | 同上。誤りの内容を明示して判断を仰ぐ |
| A-1〜A-6 該当事象 | `user-confirmation-minimization.md` §1 に従い報告 |
| ユーザーが明示的にプロセス詳細を求めた | 求められた範囲で補足する |

上記以外では、完了報告にプロセス詳細を混ぜない。

---

## 5. 完了・成功の定義

- [ ] 完了報告の冒頭に「ご依頼（初回指示の再掲）」がある
- [ ] アウトカム（結果）が主役で、プロセス（マージ手順・レビュー往復）が主役になっていない
- [ ] PR 番号・URL・ブランチは末尾の補足1行に収まっている
- [ ] Slack `--outcome` も初回指示アウトカム1文になっている
