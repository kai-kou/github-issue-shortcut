# リサーチ品質基準（research-rules.md）

> `research-runner` スキルと `tools/research_schema.json` が参照する Deep Research の品質基準。
> プロジェクト固有の追加基準（ドメイン用語・対象範囲・除外条件など）は本ファイルに追記する。
> タスク依存ルール（スキル起動時に Read）であり常駐は不要のため、`.claude/rules/` への symlink は作らない。

## 1. 情報ランク（A / B / C）

全ての出典（`sources[].rank`）と未確認事項（`fact_check_flags[].rank`）に以下のランクを付与する。

| ランク | 定義 | 例 |
|--------|------|----|
| **A** | 一次情報・公式発表・査読済み | 公式ドキュメント・公式ブログ・SEC/政府資料・論文・公式 API レスポンス |
| **B** | 信頼できる二次情報 | 大手報道・著名な技術メディア・公式に近い解説記事 |
| **C** | 要検証の二次/三次情報 | 個人ブログ・匿名情報・SNS・出典不明のまとめ |

> 事実主張には本文中に `[A]` / `[B]` / `[C]` タグと出典 URL を併記する（`sections[].body_markdown`）。

## 2. 正式名称の確認（official_names）

製品・人物・企業・技術の名称は **公式表記** を確認し `official_names[]` に列挙する。
- `term`（確認対象）/ `official`（公式の正式表記）/ `japanese`（日本語表記・任意）/ `source_id`（根拠）
- 通称・略称で断定せず、公式ソース（ランク A）で裏取りする。

## 3. 必須セクション・出典数

| 項目 | 基準 |
|------|------|
| セクション数（`sections`） | **5〜7**（プロンプトの調査項目に対応） |
| 各セクション本文 | 100 文字以上・`[A/B/C]` タグと URL 併記 |
| 出典数（`sources`） | **8 件以上** |

## 4. fact_check_flags（未確認・要レビュー事項）

確証が取れない主張・要レビュー事項は `fact_check_flags[]` に明示する（`claim` / `rank` / `reason` / `source_id`）。

**品質ゲート閾値**:

| 条件 | 判定 |
|------|------|
| `fact_check_flags < 5 件` かつ `rank C を含まない` | 合格 → PR へ |
| `fact_check_flags >= 5 件` または `rank C >= 1 件` | 再リサーチ（Gemini で 1回再試行） |
| 再試行後も未達 | `status:waiting-user` に戻して通知（境界外） |

## 5. 未確証の扱い

- 数値・日付・固有名は推測で断定しない。確証が無ければ `fact_check_flags` に落とすか「未確認」と明記する。
- 古い可能性のある情報は最新の一次情報で再確認する（CP-2）。

## 6. 出力フォーマット

- 構造化 JSON は `tools/research_schema.json` 準拠（`research-runner` がエンジン出力を正規化する）。

## 7. 解空間の網羅（探索フレーミング・アンカリング回避）

リサーチ／提案の観点を立てる **前** に、解空間を軸で構造化して網羅性を担保する。ユーザーの問題文の語彙にそのまま引っ張られると、有力だが言及されなかった手段（古典的だが強力な手段ほど）を取りこぼす。

### 7.1 解空間の軸出し（観点設計の前に必須）

- 観点（リサーチエージェントの担当）を決める前に、解の **次元（軸）** を 3〜5 個列挙する。
  例（capture 系）: 着地先 × 送信トリガー × 必要ツール（OS/ブラウザネイティブ｜専用アプリ｜サーバ）× 対象（ブラウザ内｜他アプリ）。
- 軸の **直積でセル** を作り、空白セルを炙り出す。空白セルが見落とし候補。
- **「追加ツールゼロ／ブラウザ・OS 標準機能だけで実現する経路」を必ず 1 軸に立てる**（automation を外部ツール前提で考えがちな偏りの是正）。

### 7.2 completeness critic（欠落探索の専任役）

- fan-out リサーチ／議論型レビューに、**「提案に *無い* 有力手段は何か」を専任で問うエージェントを 1 枚** 常設する。
- プロンプト例: 「この提案で **見落とされている** 手段、特に **枯れているが見落とされがちな古典手段**（ブックマークレット・カスタム検索エンジン・OS 標準機能・メール等）を、出典付きで挙げよ」。
- 議論型レビュー（敵対的検証）は **既存案の精査** に偏り新規選択肢を発見しない。completeness critic はこれを補完する別役割（`docs/rules/discussion-whiteboard-rules.md` の Blackboard と併用）。

### 7.3 制約次元の先出し確認（Think Before Coding）

- 解空間を絞る前に、ユーザーの **制約次元** を 1 問で引き出す（または両端を必ず探索する）:
  外部ツール可否 ／ 完結範囲（端末内のみか）／ 主な利用経路。
- 後出し制約で初めて最適解に到達する事態を防ぐ。これは A-1〜A-6 の確認ではなく「仕様確定」（`user-confirmation-minimization.md` §3 item 0）。
- 併せて人間可読の Markdown（`content/research/{ID}_deep_research.md`）を出力する。
