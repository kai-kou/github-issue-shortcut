## L-059: ファイル拡張子・メタ情報を信用せず実体を検証する（2026-06-13）

**パターン**: ファイル名やメタデータが「PNG」と言っていても実体は JPEG、といった不一致を
そのまま信用して処理を進め、後段の変換・アップロードで失敗する。

**根本原因**: 宣言された形式（拡張子・メタデータ）と実体が乖離していることを検証していない。

**対策**: 障害調査時はファイル形式を実体で確認する。
```bash
file path/to/asset   # 実際の形式を確認（拡張子と一致するか）
```
形式不一致を検出したら、変換・リネームで正規化してから後段に渡す。
`problem-investigation-protocol.md` Step 1（状況の精密化）の一部として実施する。

## L-060: リサーチの解空間を問題文の語彙にアンカリングして有力手段を取りこぼす（2026-06-28）

**パターン**: ユーザーの問題記述の語彙にリサーチ観点が引っ張られ、解空間を狭くフレーミングする。結果、
言及されなかったが有力な手段（特に枯れた古典手段）を初回提案で取りこぼし、ユーザーの後出し制約で初めて到達する。

**対策**: 観点を立てる前に解空間を軸で構造化（morphological analysis）し直積で空白セルを炙り出す。
「追加ツールゼロ／ブラウザ・OS ネイティブ」を必須軸にする。fan-out／議論に completeness critic
（欠落探索専任）を 1 枚常設する。制約次元（ツール可否・完結範囲・利用経路）を先に確認する。
詳細は `docs/rules/research-rules.md` §7。
