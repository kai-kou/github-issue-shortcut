# セッション安全ルール（ユーザー確認前コミット）

クラウド環境（Claude Web Scheduled Tasks / 手動セッション）では、セッションタイムアウトが頻発する。
ユーザーへの確認待ち中にタイムアウトが発生すると、未コミットの作業が失われセッション再開時の環境リセットで消える。

---

## 🔴 ユーザー確認に回す前の必須リサーチ義務（L-077・最重要）

**「ユーザー確認」を選ぶ前に、必ず `docs/rules/problem-investigation-protocol.md` の専門チーム調査プロトコル（5ステップ）を完全実施する。**

過去のユーザー実体験では、「環境変数がない」「API認証失敗」「ファイル不在」等を理由にユーザー確認に回したケースの大半が、 **詳細リサーチさえすれば自己解決できた** 事例だった。これは CP-6（ユーザー介入最小化）の重大違反。

### 必須プロトコル（リサーチ尽くしたか？を自問する）

```
[ ] Step 1: 状況の精密化（エラー全文・再現手順を3行で記述）
[ ] Step 2: 既存ナレッジ検索（lessons-core.md / lessons/*.md / 過去 Issue / SKILL.md を grep）
[ ] Step 3: 専門チーム並列調査（Agent Teams で コードベース・ドキュメント・公式情報の3役以上を起動）
[ ] Step 4: 3層因果分析（直接原因 / 中間原因 / 根本原因）
[ ] Step 5: 自己解決可否の判定（境界外リスト or 公式仕様上ユーザー操作必須に該当するか）
```

上記5ステップが未実施の状態で「ユーザー確認」セクション以降に進むのは禁止。詳細は `docs/rules/problem-investigation-protocol.md` を参照。

## タイムアウトの種類と対策（3 種類）

クラウド環境のタイムアウトは **3 種類** が存在し、原因・対策が異なる。

| 種類 | 発生タイミング | 症状 | 対策 |
|------|-------------|------|------|
| **セッション切れ** | スケジュールタスク・長時間アイドル | 新セッションが起動、未コミットが消える | Stop フック自動コミット・チェックポイント |
| **Request timed out** | 手動セッション中・ツール呼び出し数過多 | 「Request timed out」表示、セッション再開ボタン | 1ターンのツール上限・リモート git 対策（下記） |
| **Stream idle timeout** | 大量ファイル読み込み後の処理中 | 「API Error: Stream idle timeout - partial response received」 | Read の limit をコンテンツ種別で調整（シンプルな Markdown: max 200行、構造化: max 80行、コード: max 60行）（下記 L-055 対策） |

## ユーザー手動セッションでの Request timed out 防止ルール

### ルール1: 1 ターンのツール呼び出しは 8 個以内

ブラウザと Claude API 間の HTTP タイムアウト（約 2〜5 分）は、1 ターンで多数のツールを連続実行すると発生する。

```
上限: 1 応答ターンで実行するツール呼び出しは最大 8 個まで

超える場合:
1. ここまでの作業を中間報告（テキスト応答）として送信
2. 「続けますにゃ」等として次ターンで再開
```

**判断基準**: ファイル読み込み・編集・Bash コマンド・GitHub API 呼び出しを問わず、全ツール呼び出しの合計でカウントする。

### ルール2: セッション再開直後はリモート git コマンドを避ける

セッション切れ後のリモート git 操作は、前セッションのプロキシ接続が無効になっており **ハングする** 場合がある。

```bash
# NG: セッション再開直後のリモート通信（ハングの可能性）
git rev-parse origin/{ブランチ名}
git fetch origin {ブランチ名}

# OK: ローカル情報のみで確認する（ネットワーク不要）
git status                          # ローカル状態確認
git log --oneline -3                # 最新コミット確認
git rev-parse "@{u}" 2>/dev/null    # 追跡ブランチのローカルキャッシュ参照

# リモート情報が必要な場合: タイムアウト付きで実行
timeout 10s git fetch origin {ブランチ名} 2>/dev/null || echo "fetch failed, using local cache"
```

### ルール3: 長いタスクは「8 ツール → 中間報告 → 続き」のリズムで進める

多ステップタスク（PR トリアージ・パイプライン実行等）では自然な区切りを意識する:

```
良い進め方:
  ターン1: Issue 調査 × 3 + ラベル更新 × 3 + コメント投稿 × 2 = 8 tools
           → 「Phase 1 完了。次は PR 作成に進みますにゃ」
  ターン2: PR 作成 + レビュー依頼 + subscribe = 3 tools → 完了報告

悪い進め方:
  1ターンで Issue 調査 + ラベル × 5 + コメント × 3 + PR + レビュー + ... = 14 tools
  → Request timed out
```

### ルール4: Read ツールの limit はコンテンツ種別で調整する（最大 200 行・複雑な構造化コンテンツは 60〜80 行）（L-055 対策）

大量の複雑なコンテンツ（Python コードブロック・YAML テーブル多数）を1回 Read で取得すると、処理中にストリームアイドルタイムアウトが発生する。

```
limit の目安（コンテンツの複雑さに応じて調整）:
  - シンプルなテキスト・Markdown: max 200行
  - コードブロック・テーブルが多い構造化 Markdown: max 80行
  - Python スクリプト・大量コード: max 60行

良い例:
  Read(file, offset=96, limit=80)    # 1回目（複雑な構造化 Markdown）
  → 即座に短いテキスト応答「1/3 読み込み完了。次の80行を続けるにゃ」
  Read(file, offset=176, limit=80)   # 2回目（次ターンで）

悪い例:
  Read(file, offset=96, limit=400)   # 378行を一度に取得 → Stream idle timeout
```

**重要①**: **大きな Read（複雑な構造化 Markdown 80 行超 / コード 60 行超）の返値を受け取った直後** に限り、次のツール呼び出しより前に **短い事実 1 行** を出力する。サイレントに次の処理を計画すると Adaptive Thinking が長くなり stream idle になるのを防ぐためで、目的は **タイムアウト回避** のみ。

```
良い例: Read(...) → テキスト「内容確認したにゃ。次はXXするにゃ」→ Write(...)
悪い例: Read(...) → [サイレント思考] → Write(...)  ← stream idle timeout
```

> **注意（verbosity との両立・`output-verbosity-rules.md`）**: 本ルールは **大きな Read 直後の短い事実 1 行** に限る。これを口実に、内部検証・テスト・探索の **毎ツールの分析的実況**（`Test 1 ✓ …` 等）を本文に垂れ流してはならない（トークン浪費・ユーザーには追えない）。小さな Read や grep/test 等の戻り値の逐次ナレーションは不要。内部作業はサイレントに実行し、統合アウトカムを 1 回で報告する（`docs/rules/output-verbosity-rules.md` ・ L-111）。

**重要②**: Read 後に大きなファイル（200行超）を生成する予定がある場合、Read を省略して既知の内容から直接 Write する。

**圧縮サマリーの活用**: コンパクション後のサマリーにファイル内容が記録されている場合、再 Read せずサマリーを参照する（再読み込みによる二重タイムアウトリスクを避ける）。

### ルール5: ツール出力の空結果・欠落は retry + クロスチェックしてから信じる（#81 対策）

クラウド環境では Bash / Read / GitHub MCP / ToolSearch が **断続的に空結果・欠落を返す** ことがある（プロキシ・レンダリング不安定）。これを「本当にゼロ件」と即断すると完了判定を誤る（実際にはマージされていない PR を「マージ済み」と誤認する等・本ベースで実際に発生）。

- 状態確認系コマンド（`mcp__github__*` / `git log`、ローカルでは `gh pr list --json number`）がゼロ件・空を返したら、**即座にリトライ**（短い待機 → 最大 3 回）する。
- 特に **完了判定・存在確認に使う結果** は、単一コマンドの戻り値を鵜呑みにせず、別経路（`mcp__github__*` と `git log`、ローカルでは `gh` も併用）で **クロスチェック** してから判断する（クラウドの gh は 403 のためクロスチェック経路にならない・L-114）。
- 3 回超リトライしても安定しないときに初めて確認に回す（L-079 のフォールバック順と整合）。

## git 操作の安全則（マージ後の ref 同期・commit 着地確認・完了検証）

クラウド環境のプロキシ不安定・squash マージの特性により、git 操作は「成功したつもり」で着地していないことがある。以下を機械的に守る。

### G-1: squash マージ後は origin/main を明示 refspec で同期してから新ブランチを切る（#78）

squash マージはマージコミットを作らないため、ローカルの `origin/main` が古いまま残ると、そこから切った新ブランチの merge-base がマージ前コミットになり **過去 PR の差分が二重に出る**（手戻りの常習原因）。

```bash
# マージ後・新ブランチ作成前に必ず実行（remote-tracking ref を確実に更新）
git fetch origin +main:refs/remotes/origin/main   # +で非fast-forward（force push等）にも追従
git rev-parse --short origin/main         # 期待: 直近マージのコミット
git checkout -B <new-branch> origin/main  # 最新の origin/main を起点にする
```

ハーネスでも `session-start.sh` / `pre-compact.sh` / `post-compact.sh` が同じ明示 refspec で fetch するため、素の `git fetch origin <branch>` による古い remote-tracking ref 残留を防ぐ。

### G-2: commit はヒアドキュメント `-m "$(cat <<EOF)"` を避け、着地を確認する（#77）

`git commit -m "$(cat <<'EOF' ... EOF)"` はシェル残響でコミットが空振りし、HEAD が進まないことがある（本ベースで実際に発生）。**複数 `-m` フラグ** を既定にする。

```bash
git commit -m "件名" -m "本文1行目" -m "本文2行目"
git diff HEAD~1 HEAD --stat   # 直後に差分が記録されたか確認（空なら commit は着地していない）
```

### G-3: 完了報告の前に実マージを機械検証する（#77）

PR 番号の存在だけで「マージ済み」と判断しない。完了報告の **直前** に必ず:

```bash
git fetch origin +main:refs/remotes/origin/main   # +で非fast-forward（force push等）にも追従
git log origin/main --oneline | head -3                  # 自分のマージコミットが載っているか
# state=MERGED の確認（クラウド一次経路・L-114）: mcp__github__pull_request_read(method="get", pullNumber=N)
# ローカル: gh pr view <N> -R <owner>/<repo> --json state,mergedAt
```

`state=MERGED` を確認できるまで「マージ済み」と報告しない（クラウドの gh は 403 のため MCP で確認する・ルール5）。

## 基本ルール

**タスク完了以外でユーザーへの確認・判断・報告を求める前に、未コミット・未追跡ファイルをすべてコミット＆push する。**

例外（コミット不要・確認メッセージをそのまま送ってよいケース）:
- タスク完了時の最終報告（「PR作成しました。レビューをお願いします」等）
- git 管理外の作業のみの場合（Issue コメント投稿・ラベル更新のみ等）

## ユーザー確認前の手順

```bash
# Step 1: 状態確認
git status

# Step 2: 未追跡・未コミットがある場合
git add .                        # カレントディレクトリ以下の変更を全てステージング
git commit -m "[V{ID}] wip: ユーザー確認待ち（{概要}）"
git push -u origin {ブランチ名}

# Step 3: push 完了後にユーザーへの確認メッセージを送る
```

### コミットメッセージの例

| 状況 | コミットメッセージ例 |
|------|------------------|
| fact_check_flags が多い | `[V003] wip: ユーザー確認待ち（fact_check_flags 7件・要重点レビュー）` |
| 想定外の問題が発生 | `[V003] wip: 中断（VOICEVOX接続エラー。ユーザー確認待ち）` |
| 品質ゲート未達 | `[V003] wip: 音声尺チェック待ち（実測7分32秒・最低尺8分未満）` |
| アーキテクチャ判断が必要 | `[V003] wip: 設計方針確認待ち（セクション構成を変更する必要あり）` |

## コミットできない場合

| 状況 | 対応 |
|------|------|
| main ブランチにいる | 新しいブランチを作成してからコミット（`git checkout -b claude/wip-{概要}`） |
| 変更がない（git status が clean） | そのままユーザーへの確認メッセージを送ってよい |
| push がネットワークエラー | リトライ最大4回（指数バックオフ）後も失敗したら Issue コメントに作業内容を記録 |

## 確認が必要になるケース一覧

> **SSOT 参照**: 「どのケースが真にユーザー確認を要するか」の判定は `docs/rules/user-confirmation-minimization.md` が正本。確認に回す前に同 §3 のエスカレーション前必須チェックリスト（8項目）を満たすこと。A/B/C/D 分類で A（既約境界外）に一致しない確認は原則 CP-6 違反。

以下のケースでユーザー確認が発生する。いずれもコミット後に確認を求める。

### 脱線検知時

- 想定外のエラーやブロッカーに遭遇した場合
- 想定の方法では解決できないことが判明した場合

> **必須**: ユーザー確認に回す前に `docs/rules/problem-investigation-protocol.md` の専門チーム調査プロトコルを完全実施する（L-077）。リサーチ尽くした上で「自己解決不可」と判定された場合のみ確認に回す。

### 品質ゲート未達時

| ゲート | 基準 | 対応 |
|--------|------|------|
| 音声尺不足 | VOICEVOX 実測 8 分未満 | 台本根本見直し。ユーザーに報告 |
| fact_check_flags 過多 | 5 件以上残存 | PR 説明文に記載。ユーザーに注意喚起 |
| セルフレビュー Error | 残存 | PR 作成中断。ユーザーに報告 |
| AIレビュー指摘 未収束 | 2 サイクル超 | サーキットブレーカー発動。ユーザーに報告 |

### パイプライン停止時

`/script-pipeline`, `/audio-pipeline`, `/video-pipeline` 等の自動パイプライン内で停止が必要な場合:

1. その時点のファイルをコミット
2. **Issue / PR コメントに「次に再開すべきステップ」を必ず記録する**（下記テンプレート参照）
3. ユーザーへの確認メッセージを送る

### Issue コメント記録テンプレート（パイプライン停止時）

```
## 🔴 パイプライン停止（{理由}）

**停止ステップ**: Step {N} / {フェーズ名}
**停止時刻**: {YYYY-MM-DD HH:MM} JST
**ブランチ**: `{ブランチ名}`
**最新コミット**: `{commit_sha}`

### 完了済み
- [x] Step 1: {完了内容}
- [x] Step 2: {完了内容}

### 次回再開ステップ
- [ ] Step {N}: {次に実行すべき具体的なアクション}
  - コマンド例: `{実行コマンド}`
  - 必要なファイル: `{パス}`

### 停止理由
{具体的な理由と確認が必要な事項}

### 実施済み調査（problem-investigation-protocol.md 必須）
- 既存ナレッジ検索: lessons / Issue / SKILL.md を {N} 件確認 → 該当 {あり/なし}
- 専門チーム並列調査: {役割と調査結果サマリ}
- 試行した解決策: {時系列リスト}
- 特定した直接原因 / 中間原因 / 根本原因: {3層分析の結果}
- ユーザー判断が必要な理由: [ ] 境界外リスト該当 / [ ] 公式仕様上ユーザー操作必須 / [ ] サーキットブレーカー発動

### 提案する選択肢
1. **推奨**: {案A}
2. {案B}
3. 中止: {案C}
```

**記録先の優先順位**:
1. 対象動画の制作 Issue（`[V{ID}] Phase {N}: {概要}` 形式の Issue）
2. 対象 PR のコメント（PR が既に作成済みの場合）
3. なければ一般的な wip Issue を作成してコメント

## パイプラインスキルでの適用

各パイプラインスキルに「ユーザーに報告する」ステップがある場合、必ずその直前にコミットを実施する。

```
（パイプライン内の停止パターン）

X. エラー検知・品質ゲート未達
  ↓
X.1: git status → 未コミットを確認
  ↓
X.2: git add → git commit → git push
  ↓
X.3: ユーザーへの報告メッセージ（停止理由・推奨アクション）
```

## セッション再開プロトコル（自律実行・ユーザー指示不要）

**「続きをやって」等のユーザー指示を待たず**、セッション開始時に自律的に以下を実行する。

### Step 1: 状態確認（並列実行）

```bash
# 現在のブランチと最新コミットを確認
git branch --show-current
git log --oneline -5

# レビュー待ちPRを確認（① 自セッション作成 PR を最優先で回収 → ② 共有スコープで孤児救済・#47）
python3 tools/check_pending_pr_reviews.py --mine --actionable-only --json
python3 tools/check_pending_pr_reviews.py --actionable-only --json

# 進行中/待機中のIssueを確認
# クラウド（一次経路・L-114）: mcp__github__list_issues(labels=["status:in-progress"], state="OPEN") /
#   mcp__github__list_issues(labels=["status:waiting-claude"], state="OPEN")
# ローカル:
gh issue list -R kai-kou/github-issue-shortcut \
  --label "status:in-progress" --state open --limit 10
gh issue list -R kai-kou/github-issue-shortcut \
  --label "status:waiting-claude" --state open --limit 10
```

### Step 2: 作業内容の復元

1. `[wip]` または `wip:` のコミットメッセージがある → 前セッションで途中だった作業が存在
2. Issue コメントの「次回再開ステップ」を探して前回の停止箇所を特定する
3. `status:in-progress` のIssueがあれば、そのIssueの最新コメントを確認する

### Step 3: 自律的に作業再開

- **「やってよいですか？」と聞かない**。CP-1（自律的判断と行動）に従い、そのまま再開する
- レビュー待ちPRがあれば `pr-review-watcher` フローから再開する
- `status:waiting-claude` のIssueがあれば優先度順に対応する
- `[wip]` コミットが最新なら、そのコミット内容を元に続きから実施する

### 例外（再開せずユーザーに確認が必要な場合）

- `status:waiting-user` のIssueがある → ユーザーのアクション待ちのため作業できない
- 複数の並行作業が競合しそうな場合 → どれを優先するかユーザーに確認する
- 前回の停止理由が「設計判断が必要」等の高難度判断の場合 → 内容を報告してから判断を仰ぐ
