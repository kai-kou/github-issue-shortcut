# セッションスプリント運用ルール（Session-Sprint Rules・SSOT）

> **このファイルは「1 セッション = 1 スプリント」運用の唯一の正本（Single Source of Truth）である。**
> 飼い主の明示決定（2026-06-12・Issue #2946。スクラム提案 `docs/proposals/scrum-workflow-adaptation.md` への回答）に基づき新設。
> `.claude/rules/` に symlink され、全セッションで自動読み込みされる。
>
> **飼い主決定の記録（2026-06-12）**:
> 1. **1 セッション = 1 スプリント** とみなし AI Agent に特化させる
> 2. **PO（プロダクトオーナー）は @owner ロールを拡張** して与える（**実行権限付き** — AskUserQuestion で確認済み）
> 3. **スプリント報告は日次**。1 セッションあたりのベロシティ、SP あたりの処理時間・トークン数に注目して調整する。週次（月曜 M5）は **較正専用** で残す

---

## 1. 原則: 1 セッション = 1 スプリント

- 全セッション（hourly スロット・手動・スケジュール）は **1 回のスプリント** として扱う
- スプリントの構成: Planning（ゴール宣言）→ 実行 → Review（既存 pr-review-flow）→ Retro（SP 実績記録・既存 retrospective）
- 既存の即終了機構は維持する。**対象がないセッションは no-op スプリント**（理由を 1 行記録して終了。儀式を強制しない）

## 2. スプリントプランニング（条件付き宣言）

対象 Issue が存在する場合のみ、**`status:in-progress` ロック取得と同時に** 対象 Issue へ以下のコメントを 1 件投稿する:

```markdown
## 🏃 Session Sprint Planning
- **ゴール**: {1 文。例: 対象 {ID} の処理を完了し PR マージまで}
- **対象**: #{Issue 番号}（sp:{N}）
- **編成**: {メイン + サブエージェント役割。例: 担当パイプライン + Agent Teams レビュー 3 役}
```

- ゴールはルーティングテーブル（`hourly-routing.md`）の担当タスク + discover 結果から機械的に組み立てる（追加判断は不要）
- 対象 Issue に `sp:N` ラベルがない場合は、§3 の基準で **宣言時に付与** する
- PR 本文には `Sprint Goal:` 1 行と `sp:N`、および **`Session-Id: {UUID}`** を記載する（UUID は `echo $CLAUDE_CODE_SESSION_ID` で取得）。`Session-Id:` は **二重用途**: ① `tools/sprint_session_metrics.py` が PR↔セッションを突合してセッション単位の tokens/sp・$/sp を算出（#3024）、② `tools/check_pending_pr_reviews.py --mine` が **自セッション作成 PR の所有判定**（再起動・圧縮後も自 PR を識別してマージまで責任継続・#47）。どちらも必須。記載漏れは時間窓フォールバックで救済されるが並行セッション時は帰属不明・所有特定不能になる

## 3. SP（ストーリーポイント）

### 3.1 スケール（複雑性 × レビュー負荷 × リスク。人間の労力ではない）

| ラベル | 意味 | 目安 |
|--------|------|------|
| `sp:1` | 自明な修正・データ更新 | 1 ファイル・機械的変更（状態ファイル更新 PR 等） |
| `sp:2` | 小さな改善・単一スキルの軽修正 | retro-try の typo 級 Try |
| `sp:3` | 標準タスク | 単一パイプライン 1 工程実行・中規模ルール追加 |
| `sp:5` | 複合タスク | 複数ファイル改修・新ツール追加・要 Agent Teams 議論 |
| `sp:8` | 大型タスク | 新スキル設計・アーキテクチャ変更（**分割を検討すべきシグナル**。8 超は付けず分割する） |

> §3.1 の「目安」列はファイル数・ステップ数（= Component / Coordinative 軸）をベースに置く。AI Agent 実行ではこれに **不確実性（Dynamic 軸）の補正** を重ねる（§3.1.5）。

### 3.1.5 AI Agent 全工程実行を前提にした補正（Dynamic 軸・#45）

> 本プロジェクトは **リサーチ・判断・実装・レビューをすべて Claude Code（AI Agent）が実行** する。
> 人間チームの SP（認知的工数）とギャップが出るため、AI 実行特有の **不確実性** を補正因子として明示する。
> 専門チーム議論（claude -p 議論型・PASS）の経緯と合意は `content/discussions/sp-estimation-redesign/whiteboard.md` を参照。

**事前見積もり = §3.1 のベーススケール + Dynamic 補正** （ハイブリッド方式）:

| 補正軸 | 定義（着手前に観察可能なシグナル） | 補正 |
|--------|--------------------------------|------|
| **Dynamic（不確実性）** | 要リサーチ・clarification 見込み・仕様未確定・新規領域・探索的 | **+1〜2 SP** |

- **なぜ Dynamic 軸だけを事前補正にするか**: AI 実行のコストドライバーはファイル数（人間工数の代理指標）よりも **コンテキスト消費・ツール往復・自律ループの深さ** に強く相関する。これらは着手前に直接測れないが、**不確実性が高いほど clarification とリサーチ往復が増えてトークンが膨張する**（Anthropic の知見: 最も複雑なタスクでは clarification 頻度が最小複雑タスクの 2 倍超）。Dynamic 軸はこの非線形コストを着手前に捕捉できる唯一の事前シグナル。
- **Component（独立ステップ / ファイル数）・Coordinative（相互依存 / SSOT 同期 / 横断影響）** は §3.1 のベーススケールに暗黙に織り込み済み。独立補正軸としての精緻化は **実測 4 週分の蓄積後（フェーズ 2）に帰納的に較正** する（データゼロでの精緻化は過剰設計のため見送り・議論合意）。
- **付与例**: 「明確な仕様の単一ツール修正」= ベース `sp:3`。「要リサーチの新ツール設計（仕様探索あり）」= ベース `sp:3` + Dynamic `+2` = `sp:5`。Dynamic 補正の有無と理由は §2 のプランニング宣言または PR 本文の `Sprint Goal:` 行に 1 句で残す（事後の較正で「推定がなぜズレたか」を追える）。
- **上限の扱い（sp:8 超）**: ベース + Dynamic 補正が `sp:8` を超える場合は **キャップせず §3.1 に従いタスクを分割** する（8 超のラベルは付けない）。補正後に 8 を超えるのは、不確実性が AI Agent の capability boundary（end-to-end で完遂できる複雑性の上限）を越えるシグナルであり、分割して各片の不確実性を下げるのが正しい対処。

### 3.2 工程別標準 SP（毎回の見積もり判断を不要にする）

> 工程別の標準 SP はプロジェクト固有のため `docs/project-mission.md` に定義する（§3.1 の汎用スケール + §3.1.5 Dynamic 補正をプロジェクトの工程に当てはめる）。**暫定値で開始し、実測（§5 の tokens/sp・分/sp の中央値）で 4 週ごとに較正する** （§6・議論合意）。Dynamic 補正（+1〜2 SP）は工程によらず横断適用する。

### 3.3 付与経路（全経路で必須）

| 経路 | ルール |
|------|--------|
| refinement 昇格（バックログ Issue 作成） | `sp:3`（工程別標準） |
| ユーザー指示の Issue 化 | §3.1 基準で見積もり付与（`user-instruction-issue-rules.md`） |
| retrospective / self-improvement-loop の改善 Issue 起票 | `estimated_effort` small/medium/large → `sp:2`/`sp:3`/`sp:5` に写像 |
| 上記以外で sp なし Issue に着手するとき | プランニング宣言時に付与（§2） |

## 4. PO ロール = @owner（飼い主明示委任・2026-06-12・実行権限付き）

- **@owner はプロダクトオーナーとして、バックログの優先順位（`priority:*`）と SP 妥当性（`sp:*`）を決定し、Issue ラベル操作で直接実行する**（`mcp__github__issue_write`）
- 呼び出しタイミング: refinement 昇格時（Lv1・優先順位と sp の妥当性確認）/ 週次較正（M5・Lv2）/ バックログ滞留時・優先順位の衝突時（オンデマンド）

### 4.1 PO 権限の境界（厳守・暴走防止）

| 許可 | 禁止 |
|------|------|
| `sp:*` / `priority:*` ラベルの付与・変更 | **`status:*` ラベルの操作**（CP-4 論理ロックの破壊につながる） |
| バックログ順序・受け入れ基準の決定（コメントで明示） | Issue のクローズ・本文の書き換え（コメント追記のみ可） |
| sp 較正の提案・実行 | **A-1〜A-6 の自動承認**（従来どおり絶対禁止・`escalate_to_human` で本人へ） |
| — | ファイル編集・Bash 実行（ツールを持たない） |

> ラベルホワイトリスト（`sp:` / `priority:` のみ）は @owner 定義（`.claude/agents/owner.md`）の最優先ルールに記載。違反が観測されたら Lv3 フック昇格を検討する（`harness-escalation.md`）。

## 5. 日次スプリント報告（主役・飼い主決定 3）

- **実装**: `tools/half_day_summary.py` のスプリントメトリクスセクション（朝 07:00・夜 19:00 の既存 Slack 報告に統合。夜枠が当日の日次スプリント報告）
- **指標**（飼い主が注目する調整対象）:

| 指標 | 定義 |
|------|------|
| done_sp（ベロシティ） | 期間内マージ PR に紐づく Issue / PR の `sp:N` 合計 |
| SP あたり処理時間 | PR リードタイム（created→merged）合計 ÷ done_sp。※Issue 処理時間（in-progress→クローズ）は Pilot 2 で追加（events API が必要） |
| SP あたりコスト / トークン | `cost_log.jsonl`（cost_usd・input/output tokens）の期間合計 ÷ done_sp |
| **セッション別ベロシティ（#3024）** | `tools/sprint_session_metrics.py collect` が cost_log のセッション ID とマージ PR の `Session-Id:` トレーラーを突合し、**セッション単位** の done_sp・tokens/sp・$/sp・分/sp・稼働時間を算出（朝晩の報告に「セッション別」内訳として自動掲載） |
| no-op スプリント | discover 空振り等で成果ゼロのセッション（`run_log.jsonl` に `result: "no-op"` + `reason` を 1 行追記。担い手はパイプラインスロット自身） |

- **永続化**: cost_log.jsonl は揮発 state file のため、日次メトリクスは `content/analytics/sprint/sprint_log.jsonl`（期間合算）と `content/analytics/sprint/session_metrics.jsonl`（セッション別・同一 session_id の重複記録は `recorded_at` 最新を採用）に追記してコミット対象にする（週次較正のデータ源）
- **制約**: cost_log は実行コンテナ内の自セッション群しか見えない場合がある（クラウド環境分離）。他コンテナのセッションは帰属不明バケットに落ち、報告に「Session-Id 記載漏れ/帰属不明」として表示される（ベストエフォート）

## 6. 週次較正（月曜 07:00 M5・較正専用）

報告の主役は日次（§5）。週次では以下の **較正のみ** 行う:

1. `sprint_log.jsonl` の 7 日分から工程別の見積もり誤差を集計 → §3.2 の標準 SP を更新（プロジェクト固有の工程別 SP は `docs/project-mission.md` を更新）
1.5. `session_metrics.jsonl` の 7 日分からセッション別 tokens/sp・$/sp の分布を確認し、外れ値セッション（中央値の 3 倍超）があれば原因（モデル選択・コンテキスト肥大・レビュー往復）を 1 行記録 → 改善 Issue をデータ根拠付きで起票（**「トークン・SP・時間の最適バランス調整」の実行点・#3024**）
2. no-op 率のトレンドから構造改善 Issue をデータ根拠付きで起票
3. **形骸化シグナル検出**: プランニングコメントが定型文のみ / 較正アクションが 4 週ゼロ → 運用縮退を検討
4. waiting-claude の in/out フロー監査（2 週連続 out < in で起票元別の抑制を検討）

## 7. 禁止パターン

```
❌ 空振りスロットでプランニング宣言の儀式を行う（オーバーヘッドの方が空回り）
❌ @owner が status:* ラベルを操作する / Issue をクローズする / A-1〜A-6 を自動承認する
❌ sp ラベルなしで PR をマージする（done_sp 計測が壊れる。マージ前に付与する）
❌ 日次レポートを「出して終わり」にする（週次較正アクションに接続しない）
❌ velocity（done_sp）を性能目標・ノルマにする（見積もりインフレ＝gaming を誘発する。velocity は予測可能性の観測値であり生産性指標ではない・#45）
❌ tokens/sp・分/sp・$/sp を「下げるべき KPI」として最適化対象にする（モデル選択やレビュー省略への歪んだ圧力になる。あくまで較正用の観測値であり、目標値化しない・#45）
✅ 対象ありセッションのみ宣言 → 実行 → PR に Sprint Goal 記載 → メトリクス自動集計
```
