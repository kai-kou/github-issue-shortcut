# Deep Research プロンプト

## 調査テーマ

「思いついた瞬間」を逃さず、Android スマホのホーム画面ショートカットから数秒で特定の GitHub リポジトリに Issue を起票できるモバイル PWA（Vite + React SPA + Hono API・Cloudflare Workers 単一構成）に適した UI/UX デザインの、ノウハウ・テクニック・原則・トレンド・ベストプラクティスを調査する。目標 KPI は「起票所要時間 10 秒以内（タイトルのみなら 5 秒以内）」「起動 → 入力 → 送信の 3 タップ以内」「送信失敗時に入力内容を失わない」「初回セットアップ 5 分以内」。調査結果は、プロジェクトのデザインガイドライン・デザインルール・デザインレビュー観点（チェックリスト）・機械チェック（ハーネス / テスト）へ落とし込める形で整理する。

## 調査項目（7 項目に分解）

- **クイックキャプチャ UX パターン**: Todoist クイック追加・Google Keep・Drafts 等「最速入力ツール」に共通する設計原則（即フォーカス・単一入力・省略可能なメタデータ・キャプチャとトリアージの分離など）。「入力 1 フィールド主義」と「ラベル等の初期選択」の両立パターン
- **モバイルフォーム設計のベストプラクティス**: タップターゲットサイズ（Material Design 3 / Apple HIG / WCAG の具体値）・親指到達圏（thumb zone）・キーボード種別と autofocus・enterkeyhint・入力補助・エラー表示位置など、一次情報の具体的数値・根拠付き
- **PWA の UX ベストプラクティス**: 起動速度（app shell・Service Worker キャッシュ戦略）・スタンドアロン表示・Android ホーム画面ショートカット / App Shortcuts（manifest shortcuts）・インストール導線・Lighthouse の installability 要件
- **体感速度・パフォーマンス UX**: Core Web Vitals（LCP / INP / CLS）の 2025〜2026 年時点の閾値と計測方法・楽観的 UI（optimistic UI）・スケルトン vs スピナー・応答時間の知覚閾値（0.1 秒 / 1 秒 / 10 秒ルール）の根拠
- **失敗時 UX・入力内容の保全**: オフライン / 送信失敗時の下書き保持（localStorage / IndexedDB）・再送キュー・エラーメッセージ設計（原因 + 次のアクション提示）・GitHub API エラー（認証切れ・権限不足・レート制限）のユーザー向け表示パターン
- **アクセシビリティと片手操作**: WCAG 2.2 のモバイル関連達成基準（ターゲットサイズ・コントラスト比・フォーカス管理）・片手操作エルゴノミクス・ダークモード対応の実装原則
- **デザイン原則の機械チェック化・レビュー体制**: デザインルールを自動テスト / CI に落とす手法（Lighthouse CI・axe-core・Playwright によるタップ数 / フロー E2E 計測・バンドルサイズ予算）とデザインレビューチェックリストの構成ベストプラクティス。2025〜2026 年のモバイル UI トレンド（過剰装飾の排除・ユーティリティアプリのミニマルデザイン）も踏まえる

## 制約・前提（解空間の軸・research-rules.md §7）

- 対象は「モバイル Web（PWA）」。ネイティブ Android アプリ化（TWA 含む）は将来オプションとして言及のみで可
- 軸: 入力速度 × 表現力（ラベル・本文）× 失敗耐性 × セットアップ容易性。機能追加が起票フローに 1 タップでも追加するなら既定オフ、という判断基準を持つプロダクト
- 「追加ライブラリゼロ / ブラウザ・OS 標準機能だけで実現する経路」（standard HTML input 属性・manifest shortcuts 等）を必ず 1 軸として調べる
