# チェックポイントスキル

## 目的

長時間実行パイプライン（各パイプライン・プロジェクト定義）のフェーズ境界で **復元可能な状態スナップショット** を作成する。
`/save-session` 型のセッション保存パターンをプロジェクトのフェーズ管理に適用したもの。

## トリガー条件

- `/checkpoint` コマンドを呼び出した時
- 「チェックポイントを作って」「フェーズ境界を記録して」「セーブポイント作成」等の依頼時
- 長時間パイプラインの各 Step 完了後（L-003 対策として推奨）
- セッション切れリスクがある作業前（ユーザー確認待ち直前など）

## 実行フロー

### Step 1: 現在のブランチ・状態を確認

```bash
git branch --show-current
git status --short
git log --oneline -3
```

確認事項:
- `main`/`master` ブランチでないこと（直接 push 禁止）
- 未コミットの変更があるかどうか
- 直近のコミット内容（作業コンテキストの把握）

---

### Step 2: 変更をコミット（未コミット変更がある場合）

```bash
git add -A
git commit -m "[{ID}] checkpoint: {フェーズ名} Step {N} 完了（{概要}）"
git push -u origin {ブランチ名}
```

コミットメッセージ規約:
- `[{ID}]` は対象エンティティの ID（プロジェクト定義・例: `[V015]`）
- `checkpoint:` プレフィックスで検索可能にする
- フェーズ名: プロジェクト定義の各フェーズ名（例: `research` / `script` / `audio` / `image` / `video`）
- 概要: 何が完了したか（例: `成果物v1生成完了` / `バッチ1完了`）

変更がない場合（git status がクリーン）:
- コミットをスキップして Step 3 へ進む

---

### Step 3: 状態スナップショットを Issue コメントに記録

対象エンティティの作業 Issue（`[{ID}] {フェーズ名}: {概要}` 形式・プロジェクト定義）を検索し、チェックポイントコメントを投稿する。

```
## ✅ チェックポイント: {フェーズ名} Step {N}

**日時**: {YYYY-MM-DD HH:MM} JST
**ブランチ**: `{ブランチ名}`
**コミット**: `{commit_sha_short}`

### 完了した作業
- [x] {完了した作業1}
- [x] {完了した作業2}

### 次のステップ（セッション再開時）
- [ ] {次に実行すべきアクション}
  - コマンド: `{実行コマンド}`

### 品質チェック状態
| チェック項目 | 状態 | 備考 |
|------------|------|------|
| Done Criteria 達成確認 | ✅/⚠️/❌ | Issue 本文またはコミットに記載した完了条件を満たしているか |
| {チェック1} | ✅/⚠️/❌ | {備考} |
| {チェック2} | ✅/⚠️/❌ | {備考} |

### 失敗・問題記録（What Did NOT Work）
{試みたが失敗したアプローチと理由。なければ「なし」}
```

Issue が見つからない場合は PR コメントに投稿する。

---

### Step 3.5: ループ状態を GitHub Variables に記録（機械可読 JSON・Issue #99）

Issue コメント（人間可読）に加えて、次セッションのエージェントが自動読み込みできる
構造化 JSON を GitHub Variables に書き込む。

```bash
python3 tools/loop_state_manager.py write \
  --phase "{フェーズ名}" \
  --session-id "${CLAUDE_CODE_SESSION_ID}" \
  --loop-count 0 \
  --next-steps "{次に実行すべきコマンド1}" "{次に実行すべきコマンド2}"
```

次セッション開始時の読み込み（参考）:

```bash
python3 tools/loop_state_manager.py read
# → {"phase": "...", "session_id": "...", "next_steps": [...], "updated_at": "..."} を出力
```

書き込みに失敗した場合（gh CLI 未設定・権限不足等）は Step 4 へ進む（非ブロッキング）。

---

### Step 4: 完了報告

以下の形式でユーザーに報告する（Slack 通知は不要）:

```
チェックポイントを作成したにゃ:
- コミット: {commit_sha_short}
- Issue コメント: {issue_url} に記録済み
- 次ステップ: {次に実行すべきアクション}
```

## 利用シーン別ガイドライン

### パイプライン内の自動チェックポイント

各パイプラインスキル（プロジェクト定義）から以下のようなタイミングで自動的に `/checkpoint` を呼び出すことを推奨する:

| パイプライン（例） | チェックポイントタイミング（例） |
|-------------|--------------------------|
| 生成系パイプライン | 主要成果物の生成完了後、セルフレビュー完了後 |
| 前処理系パイプライン | 辞書・設定更新完了後、本処理完了後 |
| バッチ系パイプライン | 各バッチ完了後（L-003 対策） |
| レンダリング系パイプライン | レンダリング完了後 |

### ユーザー確認前の自動チェックポイント

`session-safety-rules.md` の「ユーザー確認前コミット」フローと連携する。
ユーザーに確認を求める直前に `/checkpoint` を実行することで、セッション切れによる作業消失を防ぐ。

## 注意事項

- `main`/`master` ブランチでは `/checkpoint` を実行しない（コミット禁止のため）
- 空コミットは作成しない（変更なしの場合は Issue コメントのみ）
- push に失敗した場合は `session-safety-rules.md` の「コミットできない場合」に従う
- チェックポイントは **作業の中断点** ではなく **継続のための記録** であるため、ブロッカーがない場合はそのまま次のステップへ進む
